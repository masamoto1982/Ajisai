Ajisai Language Specification

Status: Canonical
Version: 2026-07-15

This document is the single design authority for Ajisai. It describes Ajisai as it is. It does not record development history or transitional states. If any other document conflicts with this document, this document takes precedence.

This document is written for those implementing or porting Ajisai; users who want to use the language should start from the Reference and Playground.

Ajisai is a typed, vector-oriented dataflow language. Its safety story is the conjunction of:

  • Value-shape safety — operations check that operands have the structural shape they require (Scalar, Vector, Record, NIL, CodeBlock, handles).
  • Encoding safety — string and code values carry encoding contracts on top of their underlying fraction sequences.
  • Contract safety — every Coreword carries machine-readable contract metadata in the registry: requires and ensures clauses, partiality, NIL policy, and effect classification.
  • Bubble Rule safety — a well-formed operation that cannot produce a value projects the failure onto NIL with a structured reason, while malformed use raises an ordinary error (Section 11.2).

These layers compose. A change is conformant only if it preserves all of them.

1. Language Identity

Ajisai is an AI-first, vector-oriented, exact-real dataflow language.

Every numeric value is an exact real, defined by observation rather than by any particular representation: a number can always report a rational enclosure of itself and refine that enclosure on demand (Section 4.2). Rationals and the algebraic closure of SQRT under field arithmetic are fully decidable; general computable reals are reserved for future words. All arithmetic is exact, with no intermediate rounding, and the canonical display of a number is the nested continued-fraction form of Section 4.2.3 — a sequence derived from the value, not an internal representation. Surface numeric literals (Section 3.2) are convenience forms for the same values.

Runtime stack:

  • Rust interpreter core
  • WASM boundary
  • TypeScript GUI/runtime shell

Ajisai is designed for mechanical reasoning, automated refactoring, and structurally searchable implementation.

2.3 Semantic Firewall

Ajisai separates internal representation from observable semantics. Internal representation may change freely; observable semantics must be accessed through semantic axes and protocol fields. Machine-readable consumers must use protocol fields only. Human-readable strings are non-canonical and may change.

The following are not part of Ajisai's observable semantics:

  • Rust enum variant names
  • Rust Debug output
  • internal value representation
  • display strings
  • GUI colors
  • CSS class names
  • dictionary storage layout
  • module file layout

A consumer that needs to distinguish true, false, and unknown reads the truthValue axis and nothing else.

2.3.1 Identity Model

The Ajisai Identity Model names the layers already used by this specification. It adds no Coreword, no surface syntax, and no default behavior; it is an index that explains which existing rule answers each kind of “same?” question.

LayerNameDefinitionCanonical basis
L1Value equivalenceWhether two values are the same mathematical value. This is what EQ and the other comparison words observe.Section 4.2.4; exact algebraic equality is a decidable implementation obligation for the admitted algebraic domain.
L2Structural identityWhether the internal representation is the same, such as Rational, Algebraic, Gosper, lazy CF state, or expression-DAG shape.Section 2.3, Section 4.2.2, and Section 4.2.4: representation tags are not observable value identity.
L3Budgeted observational equivalenceWhether an equality or ordering observation is decided within its water — a bound only Tier 2 observations can hit; if not, the observation yields UNKNOWN.Section 4.2.2 and Section 7.4.1.
L4Display equivalenceWhether render(data, role) returns the same human-readable string. This is role-dependent.Section 12.1 and Section 12.2.
L5Conformance equivalenceWhether implementations agree on the conformance suite's observable stack and host-effect samples: the implementation-level meaning of “the same Ajisai”.Conformance and Identity; Section 12.3 observable-machine projections; PORTABILITY.md.

The layers have fixed implications. L2 implies L1: identical representation of the same Ajisai value is necessarily the same value. L1 does not imply L2: a rational, algebraic, or Gosper-backed value may denote the same mathematical value through different internal structure, and Corewords and external consumers must not branch on that structure. In the current algebraic domain — the admitted domain \(D\) defined in Section 4.2.7 — L1 comparisons are total and exact and never return UNKNOWN; budget exhaustion and UNKNOWN are the honest L3 outcome for future or lazy domains whose order is not settled inside the observation budget. L4 does not imply L1 and must never be used as machine evidence of equality, because display is a pure function of both data and interpretation role. L5 sits above L1–L4: an implementation conforms when its observable projections agree with the suite, while unobserved representation choices remain implementation freedom.

2.3.1.1 User-facing transparency

Users write the existing comparison words — EQ, NEQ, LT, LTE, GT, and GTE — and their existing sugar. The Identity Model explains those words; it is not a new operation. For every value constructible by the current algebraic Coreword set (the admitted domain \(D\), Section 4.2.7), comparison is exact and total, so a user can rely on examples such as 'math' IMPORT 2 MATH@SQRT 2 MATH@SQRT SUB 0 EQ evaluating to TRUE without knowing the representation or any comparison budget (the leading 'math' IMPORT brings the module into scope so the MATH@ qualified names resolve; Section 7). UNKNOWN and explicit fallback through ^ (VENT) are the visible, user-chosen escape path for future undecidable observations, not silent approximation. L2 structural identity and L4 display role are implementation and presentation concerns; they do not appear as separate user-facing equality modes.

2.6 Ajisai Minimal Core

The Ajisai Minimal Core is the vocabulary that carries Ajisai's identity: the value-space separation of absence, undecidedness, and misuse, together with the flow that carries values across the stack and dictionary. Its membership is recorded machine-readably in docs/formalization-coverage.json as the core_tier field (Section 2.3); the Minimal Core is the union of the identity and flow tiers. The identity tier is the Bubble/NIL absence domain (Sections 4.5, 11.2), the third truth value unknown (U) of the Kleene three-valued logic (Section 7.5), and structured diagnosis (Section 11) — the direct-sum value space (Section 4) whose disjoint components are kept apart and propagated rather than conflated. The flow tier is the state-transformer composition of the stack machine, the modifiers (Section 6), and dictionary resolution (Section 9). The remaining tiers are outside the Minimal Core: material (exact arithmetic, vectors and tensors, hosted modules and effects, the child runtime) and sugar (the sealed surface grammar of Section 3). The coverage source is authoritative for tier membership (Authority order, Section 2.5); this section names the boundary and fixes what it guarantees.

Minimal Core is an identity classification and cross-cuts the three established senses of “Core” in this document. It is not the Core Words dictionary tier of Section 9.3 (the permanent runtime words), not the Core Profile of the Portability Profiles (the host-independent capability set), and not the Coreword contract-metadata set of Section 7.14 (every built-in carrying a contract entry). A word's Minimal Core membership is decided by its core_tier alone, independently of those three.

Stability guarantee (normative). The observable contract of every Minimal Core word — its stack effect and its behavior under NIL pass-through (Section 7.12), under the unknown truth value (Section 7.5), and under structured diagnosis (Section 11) — is stable across versions of Ajisai. Altering the observable contract of a Minimal Core word is a breaking change to Ajisai's identity, made only by a coordinated change to this canonical document and the conformance suite (Conformance and Identity). The material and sugar tiers carry no such guarantee. They are where cultivars grow — new modules, new algorithms, alternative or self-hosted definitions of derived words — and may be added, re-implemented, or retired, provided every such word stays bound by the Minimal Core's propagation disciplines: a material word that receives NIL still passes it through, and an ordering it performs still yields the honest unknown when the observation is not settled within its water.

The comparison words EQ NEQ LT LTE GT GTE are Minimal Core by their contract, not by their algorithm. Their contract — producing a TruthValue that is true, false, or, outside the admitted domain \(D\) (Section 4.2.7), the honest unknown (Section 7.4; layer L3 of the Identity Model, Section 2.3.1) — is guaranteed; the exact-real ordering that decides them is a material concern and may change representation freely (Section 2.3; layer L2 of Section 2.3.1). The guarantee covers what a comparison observes, never how it is computed.

This section adds no Coreword, no surface syntax, no capability gate, and no execution privilege; like the Identity Model (Section 2.3.1) it names a layer already present rather than introducing a mechanism. Minimal Core membership grants a word no special runtime status and withholds none from any implementation: a faithful port or a self-hosted implementation is judged the same Ajisai by Conformance and Identity whether a given derived word is built in or defined in Ajisai. The guarantee is a commitment about what this document keeps stable, not a privileged kernel that only an official build may run.

Every numeric literal may carry an optional leading sign (+ or -); a leading + is accepted but has no effect. The accepted forms are:

  • Integer — one or more decimal digits. Leading zeros are allowed and insignificant (007 denotes 7).
  • Fraction — <digits>/<digits>. The denominator must be non-zero: 3/0 is not a valid literal and is rejected as an ordinary (malformed-literal) error. This is distinct from the runtime DIV-by-zero rule, under which the operation 3 0 DIV projects to a NIL with reason DivisionByZero (Section 11.2); a literal n/0 denotes no rational at all and never reaches that rule.
  • Decimal — an integer part, a ., and a fractional part, where at most one side may be empty: a leading-dot form (.5 and -.5) has an empty integer part, and a trailing-dot form (5.) has an empty fractional part. A bare . is not a number — it is the TOP modifier (Section 3.9).
  • Scientific notation — an integer or decimal mantissa, the exponent marker e or E, an optional exponent sign (+ or -), and one or more exponent digits (1.5e-2 1e+5 1.5E2).

All numeric literals are parsed as exact real numbers (see Section 4.2). The surface literal forms above are convenience syntax: 42 42/1 42.0 4.2e1 all produce the same internal value. Integer, fraction, decimal, and scientific-notation literals yield rationals; irrational values are produced by words such as MATH@SQRT, not by surface literals.

The nested-parentheses form ( a0 ( a1 ( a2 ... ))) is the canonical serialization and AI-readable debug form for continued fractions (Section 4.2). It is not a source-code literal: Ajisai source uses the surface forms above, and the nested form appears only in display and serialization output under the ContinuedFraction interpretation role (Section 12.2).

A string literal begins with ' and ends with the last ' before a token boundary. A token boundary is whitespace, end of input, or any special character other than ' (such as [ ] { } # = |).

Any ' that appears before a non-boundary character is a literal quote character in the string content.

3.4 Code blocks

A sequence of tokens enclosed in {...}. A code block may span multiple lines; each internal line break is preserved as a statement separator, so a multi-line block executes one source line at a time. This makes multi-line word bodies — including |-style COND bodies — writable directly, without splitting a definition across separately named words.

( and ) are not Ajisai syntactic characters. They are reserved markers — ( denotes the concept RESERVED-BEGIN and ) denotes RESERVED-END (Section 3.9) — reserved at the lexical level to prevent accidental reuse and to keep the nested continued-fraction serialization form (Section 4.2) unambiguous; encountering ( or ) in source text is a tokenizer error.

3.5 Vectors

A sequence of values enclosed in [...].

3.6 COND clauses

Inside a COND expression, clauses are separated by |. Each | clause must occupy exactly one line. This rule applies at every nesting level, so the clauses of a multi-line COND body written inside an enclosing { } are each kept on their own line.

3.7 Syntax constraints

  • All bracket pairs ([ {) must be balanced.
  • Each | COND clause must occupy exactly one line (at every nesting level).
  • The characters ( and ) are not valid in Ajisai source (Section 3.4).

3.8 Word name normalization

Word names are normalized to uppercase at runtime. add and ADD refer to the same word.

4.2 Scalar: exact-real arithmetic

All numeric values are exact reals. A scalar is defined by what can be observed of it — a rational enclosure that refines on demand (Section 4.2.2) — and every internal representation is an unobservable cost class behind that interface. The canonical observation sequence of a scalar is its regular continued fraction (CF): the integer partial quotients \([a_0;\, a_1, a_2, \dots]\), finite for rationals and infinite for irrationals, derived from the value for canonical form, display, and serialization:

\[ [a_0;\, a_1, a_2, \dots] \;=\; a_0 + \cfrac{1}{a_1 + \cfrac{1}{a_2 + \cfrac{1}{\ddots}}} \]

4.2.1 Canonical form

A continued fraction is in canonical form when:

  1. \(a_0\) is any integer.
  2. \(a_1, a_2, \dots\) are strictly positive integers.
  3. If the sequence is finite and has length \(n \ge 2\), then the last partial quotient \(a_{n-1}\) is greater than 1 (no trailing 1).
  4. The sequence terminates iff the value is rational.

These rules give every real value a unique canonical CF, so equality of canonical CFs decides equality of values whenever the comparison terminates (Section 7.4).

4.2.2 Observation contract and cost tiers

A scalar is a numeric observation process: feeding it water — the single, unified refinement budget — narrows a rational interval known to enclose the value. The contract every scalar satisfies:

  • Monotonicity. Each refined enclosure is contained in every enclosure reported before it.
  • Convergence. For every value the current Coreword set can construct, finitely much water decides the sign, the integer part, and any comparison (Section 4.2.7); refinement can accelerate those decisions but never gates their correctness. Only the reserved general-computable tier below may exhaust its water without deciding — the sole source of the logical unknown (Section 7.4.1).
  • Determinism. Equal values fed equal total water report equal enclosure sequences.
  • Non-observability. Which tier or internal form backs a value never shows through value identity, order, display, or serialization (Section 4.8).

Behind this one interface a scalar occupies one of three cost tiers. A value always occupies the cheapest tier that holds it exactly: a result that becomes rational is a Tier 0 rational, whatever tier its operands occupied.

  • Tier 0 — rational (\(\mathbb{Q}\)). A reduced fraction, stored as machine integers or arbitrary-precision integers. The enclosure is a point; every observation settles immediately, consuming no water.
  • Tier 1 — algebraic. The closure of the rationals under SQRT and field arithmetic (the admitted domain \(D\) of Section 4.2.7), carried in the multiquadratic normal form \(\sum_d c_d \sqrt{d}\). Zero-ness is decided algebraically on the normal form, so sign, order, integer part, and equality are always decidable, with no budget; enclosure refinement is a speed-up only.
  • Tier 2 — general computable real. A lazily refined, monotonically shrinking rational enclosure — the tier for values with no algebraic normal form (a future \(\pi\), \(e\), LOG, …). Its comparisons genuinely consume water and may exhaust it undecided. No word in the current Coreword set constructs a Tier 2 value; the tier is reserved for future words.

Integer state that participates in scalar identity or arithmetic uses arbitrary precision (or an equally exact strategy) at all times. Implementations must not use bounded-width numeric storage that can overflow during normal evaluation (Section 4.2.6).

4.2.3 Display and serialization

The canonical AI-readable serialization of a scalar is the nested right-associative form of its derived CF sequence (Section 4.2):

( a0 ( a1 ( a2 ... ( a_{n-1} ) ... )))

with one integer per nesting level and one closing ) per opening (. A lazy infinite CF is serialized by emitting partial quotients up to an implementation-defined display budget and terminating with the marker ...) before the unproduced quotients' closing parens; the truncated display is non-canonical and must not be parsed back as an exact value.

This nested form is not Ajisai source syntax (Section 3.4). It appears only in display and serialization output under the ContinuedFraction interpretation role (Section 12.2) and in AI-targeted diagnostics. Under the RawNumber role a rational scalar renders as a reduced numerator/denominator (Section 12.2); the surface literal style of Section 3.2 is convenience input syntax only and is not retained for display.

4.2.4 Equivalence of representations

Two scalars are equal as values iff they denote the same real number — equivalently, iff their canonical CF sequences agree at every position. The tier or internal form backing a value (Section 4.2.2) is not part of value identity and must not be branched on by Corewords or external consumers: values equal as numbers are equal however they were computed.

4.2.5 Nearest-integer continued fractions (historical note)

Historical note (non-normative). Earlier revisions of this specification defined the comparison procedure of Section 7.4.1 by streaming nearest-integer continued fraction (NICF) expansions of the two operands under a partial-quotient budget, and diagnosis.agreedPrefix counted matching NICF terms. Comparison over everything the current Coreword set can construct is now decided algebraically on the Tier 1 normal form (Sections 4.2.2, 4.2.7) — no expansion is streamed and no budget is consumed — and agreedPrefix is defined in Section 7.4.1 as the number of enclosure-refinement steps a Tier 2 comparison spent without separating its operands. NICF expansions are no longer part of the language definition; the regular continued fraction of Section 4.2.1 remains the sole canonical sequence.

4.2.6 Numeric error policy

Ajisai's exactness requirement is not limited to avoiding floating-point rounding error. A conforming implementation must not silently turn numeric work into hidden value corruption through information loss, loss of significance, cancellation, truncation error, or overflow.

  • Rounding and truncation. Core numeric operations preserve exact values. Any finite display or observation budget is an observation limit only; it is not a new approximate value and must not be parsed back as canonical data.
  • Deliberate quantization. The only operation that maps an exact value onto a coarser grid, QUANTIZE (Section 7.13), is explicit in its step and emits the exact residual alongside the quantized value; the two reconstruct the input exactly (\(q + r = x\)), so quantization is a visible, lossless-in-aggregate operation rather than a silent approximation.
  • Information loss and cancellation. Algebraically equivalent values remain represented by exact scalar structures rather than by rounded samples, so subtracting near-equal values or combining large and small terms must not discard significant information as an unreported side effect.
  • Overflow. Internal integer state that participates in scalar identity or arithmetic, including Gosper/Möbius coefficients, uses arbitrary precision or an equally exact strategy. If an implementation-specific resource limit is reached, the result is a diagnosed execution/resource failure, not wraparound.
  • Undecidable observation. When an observation genuinely cannot decide within its water — possible only when a Tier 2 value is involved (Section 4.2.2) — the result is the logical UNKNOWN (Section 7.4.1), not a rounded Boolean guess.

The design aim is therefore stronger than “rounding-error-free”: Ajisai aims for numeric flow in which every traditional error class is either absent by construction, represented exactly, or exposed through a visible diagnostic outcome.

4.2.7 Admitted exact-real domain of the current Coreword set

The value model of Section 4.2 describes exact reals in full generality, including the Tier 2 computable reals. The current Coreword set, however, produces and operates on a specific, closed sub-domain of the exact reals. Naming that domain precisely is what lets a conforming implementation guarantee that comparison is total and exact (Section 7.4) without appeal to a comparison budget, and it fixes the boundary at which a program leaves the decidable region.

Definition (admitted domain \(D\)). \(D\) is the smallest set of real numbers that

  • contains every rational (\(\mathbb{Q} \subseteq D\));
  • contains \(\sqrt{r}\) for every non-negative rational \(r\) — the values MATH@SQRT produces (Sections 4.2.2, 7.6); and
  • is closed under the four field operations \(+, -, \times, \div\) (ADD SUB MUL DIV, with division by a non-zero element).

Equivalently, \(D\) is the field \(\mathbb{Q}(\sqrt{d_1}, \sqrt{d_2}, \dots)\) generated over the rationals by the square roots of square-free positive integers — the multiquadratic closure of \(\mathbb{Q}\). Every element of \(D\) has a unique normal form as a finite \(\mathbb{Q}\)-linear combination \(\sum_d c_d \sqrt{d}\) over distinct square-free integers \(d \ge 1\) (with \(d = 1\) the rational part), because the \(\sqrt{d}\) are linearly independent over \(\mathbb{Q}\). This normal form is a representation detail (Sections 2.3, 4.2.4) and is not observable; only the value it denotes is. It is the stored form of the Tier 1 cost class of Section 4.2.2; an implementation may use any other representation so long as the observable value, order, and display agree.

Closure and boundary of the current Coreword set.

  • ADD SUB MUL DIV keep \(D\) closed: sums, differences, products, and quotients of \(D\)-elements are in \(D\). Division by an element that is exactly zero bubbles to NIL per Section 11.2; no element of \(D\) other than \(0\) is indistinguishable from \(0\), so a non-zero divisor always yields a value and the reason = undecidable DIV case (Section 7.3) cannot arise inside \(D\).
  • MOD FLOOR CEIL ROUND and the six comparison relations are total on \(D\): the sign, integer part, and order of any \(D\)-element are exactly computable in finitely many steps.
  • QUANTIZE (Section 7.13) keeps \(D\) closed and is total on \(D\) given a positive rational step: for a \(D\)-element x and rational step, both the quantized value \(q = n \cdot \mathit{step}\) and the residual \(r = x - q\) are rationals in \(D\), computed in finitely many steps.
  • MATH@SQRT is defined on the rational sub-domain only: its operand must be rational. SQRT of a non-rational \(D\)-element (for example \(\sqrt{\sqrt{2}}\)) is outside the current Coreword set and is malformed use → error (Section 11.2); SQRT of a negative rational is a well-formed domain miss → Bubble/NIL. Consequently \(D\) is not closed under SQRT, and the current Coreword set constructs no nested radicals, transcendentals, or other lazy irrationals.
  • MATH@POW is defined for an integer exponent only, which keeps the result in \(D\); a non-integer exponent is malformed use.

Because \(D\) is exactly this multiquadratic field, equality and order over \(D\) are decidable exactly and totally (Section 7.4). The water-consuming observation machinery (Section 7.4.1) exists for exact reals outside \(D\) — the Tier 2 computable reals reserved for future words (Section 4.2.2) — and its unknown outcome is unreachable for \(D\)-elements through every comparison word, COMPARE-WITHIN (Section 7.4.2) included. This is the mechanism behind the user-facing transparency guarantee of Section 2.3.1.1.

4.3 Vector

An ordered, indexable sequence of values. Vectors may be nested (tensor-like). Index base is 0. Negative indices count from the end: -1 is the last element.

Role of nesting (design intent, normative): nesting is retained for exactly two jobs, and is to be read through them wherever a design or interpretation question arises:

  • Tensor substrate. A nested Vector is the language-visible form of a multi-dimensional array. The tensor operations of Section 7.2 (SHAPE, RANK, RESHAPE, TRANSPOSE, FILL) and element-wise numeric operations over vector lanes are defined on nested Vectors, and the dense representation class of Section 4.3.1 is an optimization of exactly this use.
  • Ragged and mixed structured data. Sub-vector results (SPLIT, Section 7.1), JSON arrays (the JSON module, Section 9.1), and grouped domain structures (for example multitrack music data) require rows of differing length and mixed element types, which a flat buffer plus a shape cannot represent.

Non-goal — code as data: nesting carries no homoiconic role. Executable code is a distinct value type — the CodeBlock of Section 4.6, delimited by { } (Section 3.4) — a Vector is never executable, and code is never represented as a nested Vector. Any historical framing of vector nesting as a Lisp-style code/data structure is obsolete and must not guide the interpretation of this specification.

4.3.1 Internal representation classes

A Vector value is internally represented in one of two classes:

  • nested — a tree of Value elements (Vec<Value>). Any element type may appear, including mixed types (Scalars, Vectors, Strings, NIL, etc.).
  • dense — a SIMD-oriented DenseTensor backed by Structure-of-Arrays numerator and denominator buffers, a shape, and a validity mask. Every valid lane is an exact small Fraction; an invalid lane represents NIL occupancy without rebuilding the dense representation into nested Vec<Value> form.

The class is chosen at construction time. Observable semantics — display, ordering, equality, NIL-ness, the results of SHAPE and of LENGTH, indexing, and iteration order — are identical between the two classes. Operations are free to take a fast path when the input is dense.

Dense Tensor exactness rule:

  • Lanes admit only scalars whose canonical CF (Section 4.2) is finite and whose equivalent reduced rational fits within (i64 numerator, i64 denominator). These are stored in normalized form.
  • Scalars whose CF is infinite (irrational), or whose rational-equivalent requires BigInt storage, are exact values but are not admitted to the small-lane DenseTensor representation. They live in the nested Vec<Value> class until a CF-capable SoA representation is introduced.
  • Implementations must not truncate, round, or otherwise approximate a scalar to fit a dense lane. Any value that does not satisfy the lane admission rule causes the construction to fall back to the nested class.

No-Rebuild Principle: a dense Vector never degrades to a nested Vector solely because a lane becomes NIL. NIL occupancy is represented by clearing the corresponding validity-mask bit. Internal diagnostic reasons for invalid lanes are stored outside the dense payload in an execution-context sparse registry keyed by tensor identity and lane index; they are not embedded in DenseTensor.

Equality across classes: a dense Vector and a nested Vector compare equal when (1) flattening the nested Vector into its leaf Fractions/NIL lanes yields the same lane sequence and validity as the dense buffer, and (2) the shape inferred from the nested Vector matches the dense Vector's shape. No language-visible word distinguishes the two classes; they are interchangeable from the user's perspective.

This dual representation exists for the Virtual Tensor Unit (VTU) data-movement optimizations (see docs/dev/virtual-tensor-unit-design.md). It does not introduce approximate numeric types: all numeric leaves remain exact reals (Section 4.2). The small-lane DenseTensor is an optimization for the rational sub-domain only; SIMD vectorization of arithmetic over irrational lanes is out of scope for this specification.

4.4 Record

A collection of named fields. Each field has a string key and an associated value. Field insertion order is preserved.

4.5 NIL

NIL represents the absence of a value. It is produced by well-formed operations that yield no meaningful result (Section 11.2).

4.5.0 Diagnostic absence metadata

NIL is a diagnostic absence value. NIL identity is separate from its reason, origin, recoverability, and diagnostic context. Surface display, equality, hashing, and serialization treat all NIL values uniformly unless a protocol explicitly asks for diagnostic metadata. The equality named here is structural equality — the reason-blind element equality used inside containers and for hashing and deduplication; the comparison word EQ passes top-level NIL operands through per Section 7.12 and never compares two NILs directly (see Section 7.4, "NIL operands and NIL equality").

For any NIL value:

  • semanticKind is absence
  • shape is absence
  • capabilities includes diagnosable and nilPassthrough
  • the human-readable display text NIL is non-canonical and must not be used for machine decisions

4.5.1 NIL passthrough

Operations classified as NIL-passthrough in Section 7 do not raise StructureError when a NIL operand is encountered. Instead, they produce NIL. The rule is uniform across consumption modes and target modes: if any operand consumed by the operation is NIL, the operation consumes its operands as it normally would and pushes a single NIL result.

NIL-passthrough applies to arithmetic, comparison, and the unary numeric rounding words (see Section 7.12). It does not apply to control-flow words, type-conversion words, IO words, or to VENT (^) itself, whose entire purpose is to react to NIL.

The intent is that pipelines propagate a Bubble/NIL through subsequent computation without crashing, so that a single ^ at the end of the pipeline can supply a fallback value.

When a NIL-passthrough operation receives one or more NIL operands, the resulting NIL inherits the reason of the leftmost NIL operand that carried a reason. This makes the cause traceable through long pipelines.

4.5.2 NIL versus Unknown

NIL and the logical truth value Unknown (U, Section 7.5) are distinct and must not be conflated. NIL is an operational absence: a diagnostic bubble that records why a value is missing (division by zero, out-of-range GET, parse failure). U is a logical undecidability: a definite member of the three-valued truth domain that records that a proposition could not be settled true or false (notably a comparison involving a Tier 2 observation that did not separate within its water, Section 7.4.1). U carries the TruthValue role and is observed as truthValue = unknown; NIL is observed as semanticKind = absence.

When NIL and U meet in the same operation, NIL takes priority. NIL carries a diagnostic reason that must be preserved (PreservesReason, Section 7.14), so the stronger, reason-bearing operational information is never erased by the logical value. Concretely, a logic word (Section 7.5) that receives both a NIL operand and a U operand applies its NIL handling and produces NIL, not U, unless an absorbing definite operand (false for AND and true for OR) settles the result first.

4.8 Cost model (performance contract)

Ajisai keeps every numeric value exact (Section 4.2), yet most values run on a fast representation. This section states, as a performance contract, which values are fast, what moves a value to a costlier representation, and when a comparison spends water. It answers the user-facing "how do I predict performance" question without changing any observable value.

Relationship to the value model (normative). The cost model constrains performance, never meaning. The internal representation of a value is not observable and is not part of value identity (Section 4.2.2, Section 4.3.1); this section does not weaken that rule. Consequently:

  • A conforming implementation MAY use any representation for a value, and MAY move a value between representations at any time, as long as the observable value, order, display, and serialization are unchanged.
  • Conformance judgments MUST NOT depend on which representation or "lane" a value took, nor on any cost counter (see "Observable cost surface" below). The performance classes named here are guidance about expected cost, not a required internal layout.
  • The exactness guarantees of Section 4.2.6 are unconditional: no cost class ever licenses rounding, truncation, silent overflow, or loss of significance.

Representation cost classes (informative-normative). The following classes describe expected cost. They are the concrete realizations of the cost tiers of Section 4.2.2 and the vector classes of Section 4.3.1.

ClassWhat lands hereExpected cost
Small-rational fast laneA scalar whose reduced rational fits machine-integer numerator and denominator (Section 4.2.2 Rational, small form; Section 4.3.1 dense-lane admission).Fixed, near machine-integer speed. Scalar–scalar arithmetic and comparison over this class may take a direct fast path.
Big-integer rationalA rational whose reduced form needs arbitrary-precision integers (Section 4.2.2 Rational, BigInt form).Exact; cost grows with the operand digit count. Not admitted to the small-lane dense tensor.
Algebraic irrational (Tier 1)A value produced by SQRT of a non-square rational or by arithmetic that consumes such a value (Section 4.2.2, Tier 1).Exact; comparison, sign, and integer part are settled through the algebraic decision procedure on the normal form (Section 4.2.7) rather than a single integer step, and a CF-form display costs per derived term.
Dense vectorA rectangular vector whose every leaf is a small-rational lane (Section 4.3.1 dense class).Structure-of-arrays buffers, eligible for SIMD and bulk kernels; fast element-wise work.
Nested vectorA vector with mixed element types, ragged rows, or big-integer / irrational leaves (Section 4.3.1 nested class).Handled element by element.

Promotion (informative). A value leaves the fast lane when it can no longer be a small rational. Each promotion preserves the exact value; only the representation, and therefore the cost, changes: a result that outgrows the machine-integer range becomes a big-integer rational; SQRT of a non-square becomes a Tier 1 algebraic irrational; arithmetic over Tier 1 operands yields a Tier 1 result unless it demotes to a rational (cheapest-tier-wins, Section 4.2.2); and a vector with a mixed, ragged, big-integer, or irrational element is nested rather than dense. By the No-Rebuild Principle (Section 4.3.1) a dense vector does not promote merely because a lane becomes NIL: the validity bit is cleared in place.

Comparison water (normative restatement). The comparison cost is fixed by Section 7.4 and Section 4.2.7 and is restated here for the cost model: over the admitted domain \(D\) — every value the current Coreword set can construct — every comparison word, COMPARE-WITHIN included, is total and exact and spends no water, because order is decided on the Tier 1 normal form (Section 4.2.7) with no expansion streamed. Water is consumed only when a Tier 2 observation (Section 4.2.2, reserved for future words) is an operand; there, one unit of water is one enclosure-refinement step, and that unit is what a starved comparison's agreedPrefix counts (Section 7.4.1).

Observable cost surface (normative). Cost is made observable only through observational counters, never through the value model. A conforming implementation MAY expose runtime cost counters — for example the scalar fast-path count, the dense/nested data-movement counters (see docs/dev/virtual-tensor-unit-design.md), and the COMPARE-WITHIN budget counters — through host tooling such as the CLI --json report and the runtime-metrics query. These counters are proxies for cost and are subject to the following rules:

  • They MUST NOT enter value identity, equality, ordering, display, serialization, or any guard/cache signature; reading them MUST NOT change any result. This is the same discipline the VTU counters already follow.
  • They are not part of the conformance surface: two conforming implementations MAY report different counts for the same program, and a program MUST NOT branch on them (there is no Coreword that reads them).
  • They are diagnostic context in the sense of Section 4.5.0: informative, host-observable, and non-canonical.

This section introduces no new value type, Coreword, or protocol field, and no second design authority (Section 16.1): it consolidates existing guarantees into a user-facing performance contract and fixes the one new rule that cost counters stay observational-only.

5. Stack

5.1 Structure

Ajisai maintains a single mutable stack of values. Execution proceeds by pushing and popping values. The stack is ordered; the most recently pushed value is the top.

5.2 Two-plane architecture

The runtime is divided into two planes:

Data plane: Holds ValueData payloads. All arithmetic, comparison, and structural operations execute entirely on the data plane. The data plane contains no display or formatting metadata.

Semantic plane: Holds interpretation roles and presentation metadata keyed by stack position. Consulted only at explicit semantic boundaries: rendering, output operations, and module side effects.

These planes are strictly separate. Semantic plane contents do not influence data plane computations.

5.3 Execution step limit

Each execution has a step budget. The default limit is 100,000 steps. Exceeding the limit raises ExecutionLimitExceeded. This is a runtime safety control, not a language semantic constraint. The host may configure this budget, raising or lowering it; because the budget is a runtime safety control rather than language semantics, conformance judgments must not depend on any particular budget value.

The step budget is one of two runtime safety controls of equal rank; the other is the native recursion-depth guard of Section 8.4, which bounds non-tail user-word recursion and raises RecursionLimitExceeded (Section 11.1). Both bound how much a program may run without being language semantics, and conformance judgments must not depend on either limit's particular value.

6. Modifiers

Modifiers precede a word and alter its execution behavior. Multiple modifiers may be combined.

6.1 Target modifiers

CanonicalSugarBehavior
TOP.The word operates on the top value(s) of the stack (default)
STAK..The word's operands are taken as a group selected by a leading count (see below)

Under STAK the word's operands are taken as a group from the stack. The top stack value is a non-negative integer count N; it is consumed first, and the N values immediately below it form the operand group (a stackUnderflow is raised if fewer than N operands are present, so 1 2 3 STAK ADD without a count fails). A binary word is left-folded over the group (1 2 3 3 STAK ADD → 6/1, 1 2 3 3 STAK SUB → -4/1). An ordering or equality comparison is evaluated as a chained predicate over adjacent operands and yields a single Boolean: LT/GT require strict monotonicity, LTE/GTE require weak monotonicity, EQ requires all operands equal, and NEQ requires every adjacent pair distinct (1 2 3 4 4 STAK LT → TRUE, 5 4 3 2 1 5 STAK LT → FALSE). Under the default EAT consumption the count and the operand group are both removed; under KEEP the operand group is retained beneath the result while the count is still consumed (1 2 3 3 STAK KEEP ADD → 1/1 2/1 3/1 6/1).

6.4 Additional syntax forms

All built-in words have English-word-based canonical names (see Section 7). The forms below are syntactic sugar that the tokenizer maps to the corresponding canonical word; either spelling is accepted in source code and behaves identically at runtime.

CanonicalSugarBehavior
FLOW~Visual pipeline separator; no runtime effect
VENT^Lazy NIL-coalescing control directive: if the stack top is non-NIL, keep it and skip the following source unit unevaluated; if it is NIL, discard it and evaluate the following source unit as the fallback (normative details below)
FORC!Overrides protection checks when redefining or deleting words that have dependents
LOOKUP?Display the definition of a word (see Section 7.8)

VENT (^) is a lazy control directive, not a stack-consuming word (normative). Like the COND clause separator | (Section 7.7.1), the source form is the ^ sugar, which the tokenizer emits as a dedicated control token; the fallback it guards is the code that follows ^ in the source, not a value already resting on the stack. The earlier wording "replace it with the next stack value" is superseded by the following contract, which the implementation realises:

  1. When ^ is reached it pops the value v on top of the stack.
  2. Non-NIL top — keep and skip. If v is not NIL, v is pushed back unchanged and the fallback that immediately follows ^ is skipped without evaluation. Exactly one source unit is skipped: a single token, or one balanced [ ] vector group, or one balanced { } block group (matched by bracket depth). The fallback is therefore evaluated only in the NIL case — an erroring or effectful fallback expression produces no error and no effect when the top is non-NIL. Examples: 5 ^ 99 → 5/1 (the 99 is skipped); 5 ^ FOO → 5/1 even though FOO is undefined, because it is never evaluated.
  3. NIL top — discard and evaluate. If v is NIL, the NIL is discarded (popped and not restored) and evaluation continues normally with the following tokens, so the fallback is evaluated and its result takes the place of the vented NIL. Examples: NIL ^ 99 → 99/1; 1 0 / ^ 99 → 99/1 (the division-by-zero Bubble is vented to the fallback).

Two consequences are normative traps a producer must respect:

  • Only one source unit is skipped. A multi-token fallback is not skipped as a whole on the non-NIL branch: 1 ^ 2 3 ADD → 4/1, because only 2 is skipped and 3 ADD then runs against the surviving 1 (\(1+3\)). To supply a multi-token fallback that is skipped atomically, wrap it in a single [ ] or { } group: 1 ^ [ 2 3 ADD ] keeps 1/1 and skips the whole group.
  • A { } fallback on the NIL branch is pushed, not run. Because the NIL branch resumes ordinary evaluation, a { } fallback is treated as a code-block literal and pushed as a CodeBlock value rather than executed: NIL ^ { 42 } leaves the code block { 42 } on the stack, not 42/1. Use a bare expression (NIL ^ 42) when a computed fallback value is intended, or follow the block with EXEC.

Only NIL triggers the fallback. The logical third value unknown (U, Section 7.4.1) is not NIL and is not vented by ^; it flows on into the three-valued logic of Section 7.5 unchanged.

7. Built-in Words

Ajisai vocabulary follows the rule: Core is permanent, Module is detachable, User is editable.

Built-in words are predefined and cannot be redefined or deleted.

  • Core words belong to the Ajisai runtime itself. They are always available and cannot be deleted, hidden, unimported, or redefined.
  • Module words belong to a module dictionary. Their definitions are built in and cannot be redefined or destructively deleted, but their visibility in the current vocabulary is controlled with IMPORT IMPORT-ONLY UNIMPORT UNIMPORT-ONLY.
  • User words belong to a user dictionary. They are editable, and deletion or redefinition is controlled by dependency checks and the force modifier where applicable.

Every built-in word has exactly one canonical home, either Core or a specific module. The canonical home determines where the implementation lives and, for module-canonical words, which module name IMPORT activates them under.

Independent of canonical home, every built-in word has a set of listings identifying which dictionary views surface the word for browsing or documentation. The Core listing view and any module listing view are not necessarily disjoint: a word may be listed in both. Such a word is called a boundary word, recognising that it has both a semantic role (Core) and a capability role (module).

Listings are presentation-only. A listing does not change name resolution, IMPORT semantics, or qualified-name resolution: bare names resolve through the Core vocabulary first and then through imported modules; MODULE@WORD resolves only to that module's canonical entries. A qualified MODULE@WORD resolves only after MODULE has been imported (Section 7.10); referencing it without a prior IMPORT raises unknownWord.

The one-line descriptions above are completed by the following stack contracts. Where a word departs from the default EAT consumption (Section 6.2) or requires a wrapped argument, the contract here is normative.

  • Inspection words retain their source. LENGTH and GET do not consume the vector they read; they push their result above the unchanged source vector (an exception to the default EAT consumption). Example: [ 1 2 3 ] LENGTH → [ 1/1 2/1 3/1 ] 3/1.
  • GET index. The index may be a bare scalar ([ 1 2 3 ] 1 GET) or wrapped in a one-element vector ([ 1 2 3 ] [ 1 ] GET); both retrieve the element at index 1.   • INSERT / REPLACE signature. These take the target vector and a two-element [ index element ] vector: [ 1 2 3 ] [ 1 5 ] REPLACE → [ 1/1 5/1 3/1 ]; [ 1 2 3 ] [ 1 9 ] INSERT → [ 1/1 9/1 2/1 3/1 ]. The flat form vector index element is not accepted.
  • RANGE signature. The bounds are supplied as a vector [ start end ], optionally [ start end step ]; the end is inclusive. Examples: [ 1 5 ] RANGE → [ 1/1 2/1 3/1 4/1 5/1 ]; [ 1 10 2 ] RANGE → [ 1/1 3/1 5/1 7/1 9/1 ]. The bare form start end RANGE is not accepted.
  • COLLECT count. Like STAK (Section 6.1), COLLECT consumes a leading non-negative integer count N from the top of the stack and gathers the N values below it into one vector: 1 2 3 3 COLLECT → [ 1/1 2/1 3/1 ].
  • CONCAT on Text. CONCAT is a vector operation; a Text operand is coerced to its code-point vector before joining, so concatenating two strings yields a numeric vector rather than Text: 'ab' 'cd' CONCAT → [ 97/1 98/1 99/1 100/1 ]. (A Text value still renders with its quotes wherever it is kept intact as a collection element; Section 12.2.)

SORT is a Canonical Module + core-listed boundary word: its canonical home is the ALGO module (Section 9.1) and it is additionally surfaced in the Core listing view, so a bare SORT resolves and ALGO@SORT names the same word. It is listed here for its vector role.

Arithmetic on scalars is exact field arithmetic over the cost tiers of Section 4.2.2: rational operands combine as reduced fractions, and any operand in the algebraic tier combines through its multiquadratic normal form (Section 4.2.7), with the result demoted to the cheapest tier that holds it — 2 MATH@SQRT 2 MATH@SQRT MUL is the rational 2/1, and \((1+\sqrt2)(\sqrt2-1)\) is the rational 1/1. FLOOR and CEIL and ROUND observe their operand until the integer part of the result is determined. These three words round to the integer grid under a fixed rule and discard the fractional part. QUANTIZE (Section 7.13) generalises them to any positive rational grid and, instead of discarding the remainder, emits it as an exact residual; FLOOR, CEIL, and ROUND are recoverable as its step-one special cases.

MOD is defined by the identity \(x - \lfloor x / y \rfloor \cdot y\) (the FLOOR-based remainder), computed exactly over the tiers. Unlike DIV, a zero divisor for MOD is not bubbled: 5 0 MOD raises a custom error with message "Modulo by zero" rather than producing NIL (the two words are deliberately asymmetric here). DIV by a divisor that is exactly zero — decided algebraically on the divisor's normal form, however the zero was built — produces NIL with reason = divisionByZero (Section 11.2 Bubble Rule). Over Tier ≤ 1 zero-ness is always decidable, so the reason = undecidable arm of this word is unreachable in the current Coreword set; it remains specified for a future Tier 2 divisor whose separation from zero starves (Section 7.4). Such an outcome is an operational absence — DIV could not produce a number — and remains a NIL; it is distinct from the logical unknown (U) of the comparison words (Section 7.4.1, Section 4.5.2).

Comparisons return a truth value with the TruthValue interpretation role: true or false, or — only when a Tier 2 observation is involved and cannot be separated within its water (see below) — unknown (U, the third value of the three-valued logic of Section 7.5).

Exactness over the admitted domain (normative). NIL operands and NIL equality (normative). The "equality … treat all NIL values uniformly" rule of Section 4.5.0 governs a different surface: structural equality — the element-wise equality used when comparing containers (and for hashing and deduplication). Inside a Vector or Record, a NIL element is an ordinary element value, and two NIL elements compare equal regardless of their diagnostic metadata: 0 0 DIV 1 COLLECT NIL 1 COLLECT EQ → TRUE even though one NIL carries reason = divisionByZero and the other is a literal NIL. Every relation is independently registered with its own Coreword contract metadata (Section 7.14), is NIL-passthrough (Section 7.12), and supports the same modifier combinations (TOP or STAK crossed with EAT or KEEP).

7.4.1 Decidability and comparison water

Comparison is decided in two regimes, by tier (Section 4.2.2):

  • Tier ≤ 1 (decidable regime). Two rational operands compare as reduced fractions. Any pair involving a Tier 1 algebraic value is decided on the multiquadratic normal form of the difference (Section 4.2.7): a non-empty normal form is provably non-zero, so equality is decided algebraically and the sign is fixed after finitely many enclosure refinements. Every comparison in this regime is total and exact — it consumes no water and can never return unknown. Everything the current Coreword set constructs falls in this regime.
  • Tier 2 (observation regime). A pair involving a Tier 2 computable real is compared by refining both operands' enclosures step by step under a water budget: disjoint enclosures decide the order; equality of computable reals is not decidable, so equal values can never be proven equal — their refinement continues until the water is exhausted.

To preserve totality, a Tier 2 comparison whose water is exhausted before the enclosures separate does not diverge or error. If the water runs out undecided:

  • the comparison produces the truth value unknown (U), observed as truthValue = unknown, with the TruthValue interpretation role;
  • U is a logical truth value, not an operational absence: the comparison does not produce a reason = undecidable NIL, and the Bubble Rule (Section 11.2) does not apply to it;
  • U flows into the three-valued logic of Section 7.5 directly. It is not a NIL, so the NIL-passthrough machinery of Section 7.12 does not carry it; instead AND OR NOT settle it according to the Kleene truth tables.

This water-and-U account applies only to operands drawn from outside the admitted domain \(D\) (Section 4.2.7) — the Tier 2 values reserved for future words. Over \(D\) every comparison is total and exact and U never arises (Section 7.4, "Exactness over the admitted domain"). For Tier 2 operands, the U outcome is required for all six relations (EQ NEQ LT LTE GT GTE) and for any downstream Coreword whose result depends on a comparison (notably MIN MAX SORT and the COND clauses whose head reduces to such a comparison); the propagation rule for those words is fixed in Section 7.4.3. The default water of the bare relations is implementation-defined and is not part of observable semantics.

The agreed-prefix — the number of enclosure-refinement steps the two observations consumed without separating before the water was exhausted — is carried on the comparison's Unknown result in the machine-readable diagnosis.agreedPrefix field (Section 4.5.0): a non-negative integer. It means "the two values remained within a common enclosure through this many refinement steps" and is the observational evidence behind the U result. It is diagnostic context only and does not change the observable truthValue.

STAK-mode ordering comparisons short-circuit on the first pair that yields U: the entire stack-mode result is U, regardless of how many subsequent pairs would have decided. (A NIL operand still short-circuits to NIL per Section 7.12; when both a NIL operand and a U-producing pair are present, NIL takes priority per Section 4.5.2.)

7.4.3 Propagation of U through comparison-dependent words

The U outcome named in Section 7.4.1 is required not only of the comparison primitives themselves but of every Coreword whose result depends on a comparison. This section fixes how U flows through the six such words — SIGN ABS MIN MAX SORT COND — so that an undecidable comparison never silently produces a wrong order, a spurious error, or a definite truth value it has not earned.

Of these six, COND is a Canonical Core word; SIGN ABS MIN and MAX are canonically owned by the MATH module and SORT by the ALGO module (Section 9.1). SORT is additionally Core-listed (Section 7.1); SIGN ABS MIN and MAX are reached as bare names only after IMPORT 'MATH', and always as MATH@SIGN MATH@ABS MATH@MIN and MATH@MAX. The U-propagation contract below is a property of each word wherever it is invoked, independent of its canonical home. (MATH@NEG, the additive inverse, is not in this set: it is pure exact arithmetic that computes -x directly on the exact representation, decides no comparison, and is therefore total over the whole numeric domain with no U outcome.)

The common rule is U-honesty: when the comparison a word relies on is undecidable, the word must surface that undecidability (as U, or, for COND, as the absence of a satisfied clause) rather than fabricate a decision. None of these words may treat U as true, as false, or as a malformed-input error.

**SIGN.** This reports the sign of one numeric operand as the scalar -1, 0, or 1 by deciding its order against 0 — the exact sign of \(x\) is the decided order of \(x\) versus 0. It accepts the full numeric domain, including Tier 1 irrationals and future Tier 2 observations (Section 4.2.2), and decides that order through the same comparison as the relations (Section 7.4.1); over the admitted domain \(D\) (Section 4.2.7) the sign is total and exact. When the order against 0 starves a Tier 2 observation's water, the result is the logical Unknown (U), observed as truthValue = unknown and carrying diagnosis.agreedPrefix (Section 4.5.0) — because the program cannot be told the sign of a value whose order against 0 is unknown. SIGN is NIL-passthrough (Section 7.12): a NIL operand yields NIL, and NIL takes priority over a U-producing comparison per Section 4.5.2. A non-numeric operand is malformed use and raises an error (Section 11.2), not U.

**ABS.** This returns the absolute value \(|x|\) of one numeric operand. It is derived from the sign and exact arithmetic: it decides the order of \(x\) against 0 through the same comparison, returns -x (the exact additive inverse) when \(x < 0\), and returns \(x\) unchanged otherwise. It therefore accepts the full numeric domain including Tier 1 irrationals and future Tier 2 observations, and over the admitted domain \(D\) (Section 4.2.7) is total and exact. When the order against 0 starves a Tier 2 observation's water, the result is the logical Unknown (U) carrying diagnosis.agreedPrefix — because the program cannot be told which of \(x\) and -x is the magnitude when the sign is unknown. ABS is NIL-passthrough (Section 7.12), with NIL taking priority over a U-producing comparison per Section 4.5.2; a non-numeric operand is malformed use and raises an error (Section 11.2), not U.

**MIN and MAX.** These select one of two (or, in sequence form, several) numeric operands by the order relation. They accept the full numeric domain, including Tier 1 irrationals and future Tier 2 observations, and decide the order through the same comparison as the relations (Section 7.4.1). When the governing comparison decides, the selected operand is returned unchanged. When a Tier 2 observation's water starves, the result is the logical Unknown (U), observed as truthValue = unknown and carrying diagnosis.agreedPrefix (Section 4.5.0) — because the program cannot be told which operand is the minimum/maximum when their order is unknown. MIN and MAX remain NIL-passthrough (Section 7.12): a NIL operand yields NIL, and NIL takes priority over a U-producing comparison per Section 4.5.2. In sequence form they short-circuit on the first undecidable pair, matching the STAK-mode rule of Section 7.4.1.

**SORT.** Sorting is a transitive cascade of pairwise order comparisons; a single undecidable pair makes the position of those elements relative to each other unknown, and the sorted order as a whole is therefore not established. When every pairwise comparison the sort requires decides, SORT returns the elements in ascending order as before. When any required comparison starves (possible only with a Tier 2 element, Section 7.4.1), SORT produces the logical Unknown (U) for the whole result, carrying diagnosis.agreedPrefix for the first undecidable pair encountered. SORT does not return a partially-sorted vector, and it does not fall back to a tie-break: a partial order is not a sort. (The earlier exact-fraction-only behavior is the decided case of this rule: finite rationals always decide, so a vector of rationals always sorts.) A NIL element is handled by SORT's existing NIL policy and takes priority over a U-producing comparison per Section 4.5.2.

**COND.** A COND clause fires when its guard evaluates to a definite true. Under three-valued logic a guard may now reduce to U (for example, a guard that is itself an undecidable comparison). A guard that yields U is not a definite true, so its clause does not fire; evaluation falls through to the next clause exactly as it would for a false guard, and ultimately to the IDLE else clause if no guard yields a definite true. A U guard is therefore neither an error nor a match: it is the K3-faithful reading of "this clause's condition could not be established." If no clause fires and there is no else clause, the existing CondExhausted outcome (Section 11) applies unchanged. This makes a U guard behave, for clause-selection purposes, like false — but the distinction is observable while the guard value is on the stack (it reads truthValue = unknown, not false), and it is only the clause-firing decision that treats "not definitely true" uniformly.

In all four words the water is the same implementation-defined constant used by the bare relations (Section 7.4.1); none of them expose or override it. COMPARE-WITHIN (Section 7.4.2) remains the only word that names a budget explicitly.

7.7.1 Stack signatures and consumption (normative)

The one-line descriptions above are completed by the following stack contracts, in the same form as the vector-word contracts of Section 7.1.1. Where a word departs from the default EAT consumption (Section 6.2) or requires a particular argument order or shape, the contract here is normative.

Two conventions apply to every higher-order word below. First, the block argument is the top-of-stack operand and may be written either as a { ... } code block or as a quoted word name (a Text value naming a defined or built-in word, e.g. '+' or 'DBL'); both are executed the same way. Second, each block runs in an isolated evaluation: the runtime saves the working stack, seeds a fresh stack with exactly the inputs named in the contract, runs the block, and reads back exactly the outputs named — any additional values the block leaves are discarded, and the saved stack is restored beneath the collected result. All of these words are not NIL-passthrough (Section 7.12): a NIL target is handled by the per-word rule below rather than propagated.

  • COND collects clauses and consumes the subject. COND first consumes all consecutive CodeBlock values on top of the stack as its clauses (taken bottom-to-top, i.e. in source order; at least one is required), then consumes one further value beneath them — the subject. The subject is consumed whether or not any clause fires; values deeper on the stack are untouched. Each clause is either { guard | body } (exactly one |) or the paired form { guard } { body }; a single COND must use one style for all clauses (mixing is an error), and |-style clauses are written one per source line.
  • COND guard evaluation and firing. Guards are evaluated in source order. Each guard runs in isolation on a stack seeded with a fresh copy of the subject, so a guard that consumes the subject (e.g. [ 2 ] EQ) does not disturb the subject seen by later guards. A guard fires iff it reduces to a definite TRUE (a bare Boolean TRUE, or a one-element vector wrapping TRUE; a bare non-zero number is also accepted as truthy and 0 as false). A guard that reduces to the logical unknown (U, Section 7.4.3) does not fire — it falls through exactly as a FALSE guard, and is neither a match nor an error.
  • COND else clause and exhaustion. A clause whose guard is exactly IDLE is the else clause; it is taken only after every non-IDLE guard has been tried and none fired (its position among the clauses does not matter). If no guard fires and there is no IDLE clause, COND raises CondExhausted (an error, not a NIL). COND is not NIL-passthrough: a NIL subject is an ordinary subject value, matched by guards and IDLE like any other.
  • COND body result. The winning clause's body runs in isolation on a stack seeded with a fresh copy of the subject. COND takes exactly one value — the top of the body's resulting stack — as its result and pushes it above the values that were beneath the subject; any other values the body leaves are discarded, and the body must leave at least one value or COND raises. Example: { { [ 2 ] EQ | [ 200 ] } / { IDLE | [ 0 ] } COND } 'CLASSIFY' DEF gives 2 CLASSIFY → [ 200/1 ] and 5 CLASSIFY → [ 0/1 ], with the subject consumed in both (9 2 CLASSIFY → 9/1 [ 200/1 ]).
  • MAP — vector block MAP → result-vector. Consumes the vector and the block. For each element, in order, the block runs on a stack seeded with just that element and must leave exactly one value; the results are collected into a new vector preserving order (a one-element vector result is unwrapped to its element). A NIL target yields NIL; a non-vector, non-NIL target is a StructureError. Example: [ 1 2 3 ] { 2 MUL } MAP → [ 2/1 4/1 6/1 ].
  • FILTER — vector block FILTER → kept-vector | NIL. The block runs per element and must leave a definite Boolean (or a truthy scalar / one-element Boolean vector); the element is kept unchanged iff the predicate is TRUE. The result is the vector of survivors, or NIL when none survive and for a NIL target. Example: [ 1 2 3 4 ] { 2 MOD 0 EQ } FILTER → [ 2/1 4/1 ].
  • FOLD — vector init block FOLD → accumulator. The accumulator argument order is vector then init then block. For each element in order, the block runs on a stack seeded with accumulator element (accumulator below, element on top) and must leave exactly one value, which becomes the new accumulator; the result is the final accumulator. A NIL target yields init unchanged. Example: [ 1 2 3 4 ] 0 { ADD } FOLD → 10/1.
  • UNFOLD — seed block UNFOLD → vector | NIL. The generator block runs on a stack seeded with the current state and must leave either NIL (stop) or a two-element vector [ element next-state ]. Each yielded element is appended and iteration continues with next-state (a NIL next-state also stops, after yielding). The result is the vector of yielded elements, or NIL if none were yielded; the initial seed is the value beneath the block. Non-termination is bounded by an implementation-defined iteration cap, which raises when exceeded. Example: a countdown generator over the state [ 1 ] yielding 1 2 3 gives [ 1/1 2/1 3/1 ].
  • ANY — vector block ANY → Boolean. TRUE iff the predicate is TRUE for at least one element (short-circuiting on the first). A NIL target yields FALSE. Example: [ 1 2 3 ] { 5 GT } ANY → FALSE.
  • ALL — vector block ALL → Boolean. TRUE iff the predicate is TRUE for every element (short-circuiting on the first non-true). A NIL target yields TRUE (vacuously). Example: [ 1 2 3 ] { 0 GT } ALL → TRUE.
  • COUNT — vector block COUNT → [ n ]. The number of elements for which the predicate is TRUE, returned as a one-element vector [ n/1 ]. A NIL target yields the bare scalar 0/1 (not wrapped). Example: [ 1 2 3 4 ] { 2 MOD 0 EQ } COUNT → [ 2/1 ].
  • SCAN — vector init block SCAN → vector. Like FOLD, but collects every intermediate accumulator (after each element) into the result vector (each collected accumulator that is a one-element vector is unwrapped); the result length equals the number of elements. A NIL target yields NIL. Example: [ 1 2 3 4 ] 0 { ADD } SCAN → [ 1/1 3/1 6/1 10/1 ].

7.13 Explicit quantization and residual

The core numeric words keep values exact (Section 4.2.6); the only operation that deliberately maps a value onto a coarser grid is QUANTIZE. It makes that projection explicit in both its step and its outcome, and — unlike FLOOR, CEIL, and ROUND — it does not discard the fraction it rounds away. The discarded amount is emitted as an exact residual, so quantization loses nothing that is not deliberately dropped afterward.

CanonicalSugarDescription
QUANTIZE—Quantize a scalar to a positive rational grid by banker's rounding, pushing the quantized value and the exact residual

Stack effect (TOP, EAT). [ x ] [ step ] -> [ q ] [ r ], where

  • x is the scalar to quantize;
  • step is a strictly positive rational quantum — for example 1 100 / for a two-decimal (cents) grid or 1 100000000 / for a satoshi;
  • q is the quantized value, an exact integer multiple \(q = n \cdot \mathit{step}\); and
  • r is the exact residual \(r = x - q\).

Residual identity. For every well-formed call

\[q + r = x \quad\text{exactly.}\]

Because x, q, and r are all exact, the pair ( q r ) reconstructs x with no loss. Discarding the residual is still possible, but it requires an explicit stack action, so silent loss of significance cannot occur.

Rounding rule. q is the multiple of step nearest to x; a tie (when x lies exactly half a step between two multiples) is resolved to the even multiple. This is banker's rounding (round-half-to-even): unlike ROUND, whose ties break away from zero, it introduces no systematic bias when many half-way values are quantized, which is why it is the appropriate default for currency. Let \(m = x / \mathit{step}\); then QUANTIZE chooses the integer \(n\) nearest \(m\), rounding a half-integer \(m\) to the even neighbour, and returns \(q = n \cdot \mathit{step}\).

Relationship to the integer-rounding words. With step equal to 1, QUANTIZE rounds to the integer grid, so x 1 QUANTIZE followed by dropping the residual computes the ties-to-even rounding of x. FLOOR, CEIL, and ROUND (Section 7.3) are the step-one directed and half-away special cases, retained for convenience and discarding their residual by definition.

NIL, error, and decidability.

  • x is NIL. The bubble passes through to both outputs, carrying its reason; q and r are each NIL (Section 7.12). The step is still required to be valid.
  • step is not a strictly positive rational (zero, negative, non-rational, or NIL). This is malformed use and raises a channel error, mirroring the deliberate asymmetry of MOD by zero (Section 7.3); it is not projected to a bubble.
  • Decidability. For x in the admitted domain \(D\) (Section 4.2.7) and rational step, both q and r are exact rationals computable in finitely many steps, so QUANTIZE is total on \(D\). For a future Tier 2 x outside \(D\) whose nearest multiple cannot be decided within the internal water, both outputs project to an Undecidable Bubble/NIL (Section 7.4.1, Section 11.2) rather than guessing.

Worked example. Quantizing the exact share 100 3 / to cents:

100 3 / 1 100 / QUANTIZE   ( -> q = 3333/100, r = 1/300 )

Here \(q + r = 3333/100 + 1/300 = 100/3\) exactly, so summing the two outputs with ADD returns the original value. Quantizing each share of a split this way and carrying the residuals is what lets an allocation of a total across several parts reconcile to the total with no invented or lost fraction of a cent.

7.14 Coreword contract metadata

Every Coreword (built-in or module-provided) has a machine-readable contract entry in the Coreword registry. The contract is the authoritative description of the word's input requirements, output guarantees, and runtime classification. Tests, documentation, and tooling consume this metadata; narrative text is non-canonical.

Terminology. In this specification a Coreword is any built-in word that carries a contract entry — that is, a Core Word or a Module Word (Section 9.3). It is therefore broader than the Core Words dictionary tier, which names only the permanent runtime words. “Coreword” (contract registry) and “Core Word” (dictionary tier) are distinct terms and must not be conflated.

It is pure and deterministic.

Comparison words (EQ NEQ LT LTE GT GTE) are Projecting: they are total because every well-shaped input yields a TruthValue result, projecting the undecidable case onto the truth value unknown (U) rather than raising. Their nil_policy is Passthrough (they pass NIL operands through per Section 7.12); when a Tier 2 observation starves they produce U, which is a TruthValue result and not a reason-bearing NIL, so they are no longer classified CreatesNil for that case. (The reason = undecidable NIL of earlier revisions is retired from the comparison path; see Section 7.4.1.)

Logic words (AND OR NOT) are Total over the three-valued truth domain of Section 7.5. Their registry nil_policy is Passthrough: they pass NIL operands through per Section 7.12. The nil_policy field holds a single value, so reason-preservation is not a second simultaneous policy here but an additional behavioral requirement layered on Passthrough — the leftmost NIL reason must survive and a NIL operand is never silently replaced by U (Section 4.5.2). The distinct PreservesReason value is reserved for words whose only NIL contract is to carry a reason through unchanged: the pure no-op and marker words TOP STAK EAT KEEP TRUE FALSE NIL IDLE FLOW VENT.

COMPARE-WITHIN (Section 7.4.2) is Projecting: it is total over well-shaped input because it projects the budget-undecided case onto the logical Unknown (a result, not a reasoned NIL). Its nil_policy is Passthrough for the \(a\) and \(b\) operands. A non-positive or non-integer budget or non-numeric operands are malformed use and raise an error, so it is not CreatesNil.

SIGN and ABS and MIN and MAX and SORT (Section 7.4.3) are Projecting: each is total over well-shaped numeric input because it projects an undecidable governing comparison onto the logical Unknown (a result, not a reasoned NIL). Their nil_policy is Passthrough, with NIL taking priority over a U-producing comparison (Section 4.5.2). SIGN and ABS and MIN and MAX are canonically MATH words and SORT is canonically an ALGO word (Sections 7.4.3, 9.1); their contracts are registry entries on the same footing as Core contracts. MATH@NEG is instead pure exact arithmetic — Total with Passthrough nil policy, no comparison and no U — computing the additive inverse directly on the exact representation. COND (Section 7.7) treats a U guard as not-firing rather than producing U as a value, so its existing partiality and CondExhausted behavior are unchanged by Section 7.4.3.

Other module-canonical words carry registry contracts on the same footing. MATH@POW is Projecting with CreatesNil: it projects 0 raised to a negative exponent onto Bubble/NIL (reason = divisionByZero) while malformed use raises an error. MATH@GCD and MATH@LCM are Partial with Passthrough: a non-integer numeric operand is malformed use and raises an error (cf. CHR), while NIL operands pass through. ALGO@INDEX-OF and TIME@PARSE-ISO are Projecting with CreatesNil, projecting a well-formed miss (value absent or unparseable text) onto Bubble/NIL with reason = missingField and reason = invalidEncoding respectively. Adding any Coreword — Core or module — without a contract entry is a conformance violation.

7.15 Diagnostic absence accessors

The five words below are Canonical Core words that read the diagnostic metadata of Section 4.5.0 from a value on the stack. They are the normative means by which a program observes why a value is absent, so that a Bubble/NIL (Section 11.2) can be triaged rather than uniformly collapsed with a single VENT (^) fallback.

Observation, not consumption. All five retain the inspected value on the stack and push their result above it, following the inspection-word precedent of Section 7.1.1 (LENGTH and GET do not consume the vector they read). A diagnosis is an observation, not a consumption. Under KEEP the retained target is unaffected; there is nothing extra to bifurcate because the target is already retained. Their static flow-mass contract (Section 13.1) is therefore Dynamic, exactly as for the retain-source inspection words — a Fixed contract would mis-model the retained operand for the static depth analyzer.

Operational NIL only (firewall). The accessors act on operational absence only. The logical truth value Unknown (U, Section 7.5) shares NIL storage but is not an operational absence (Section 4.5.2); applied to U, NIL? yields FALSE and the other four yield NIL. In particular the internal logicallyUnknown reason that U carries is never observable through NIL-REASON — this preserves the Section 2.3 / Section 7.5 firewall that forbids inferring U from semanticKind = absence or from any absence.reason.

Bubble Rule consistency. Applied to a value that is not an operational NIL, NIL? yields FALSE and each of the other four is a well-formed operation that cannot produce a value, so it yields NIL (Section 11.2). None of the five raises on a well-formed input; only malformed use — an empty stack — raises StackUnderflow. 8. User Words

8.1 Definition syntax

{ body... } 'NAME' DEF

DEF takes exactly two positional arguments: the name (a string) on top of the stack, and directly below it the body (a code block { }). Roles are determined by position alone — no value types are inspected to guess them — so a leftover value on the stack below the body cannot shift argument interpretation.

The body must be a code block. It may span multiple lines (Section 3.4): each internal line break is a statement separator, so a multi-line body executes one source line at a time. This is how multi-line and |-style COND word bodies are written:

{
  { [ 1 ] = | [ 100 ] }
  { [ 2 ] = | [ 200 ] }
  { IDLE   | [ 0 ]   } COND
} 'CLASSIFY' DEF

A data array [ ... ] is not accepted as a definition body; defining a word from data is reserved for a future data-to-code conversion word.

8.2 Rules

  • User words are stored per active dictionary (namespace).
  • Built-in words cannot be redefined.
  • A user word that has active dependents requires the force modifier ! to be redefined.
  • Dependencies are tracked automatically at definition time, by content identity rather than by name (Section 8.6).

8.3 Deletion syntax

'NAME' DEL
'DICT@NAME' DEL

Deletes a user word. The force modifier ! is required if other words depend on the word being deleted.

DEL never destroys module dictionaries or module words. To remove module words from the current vocabulary, use UNIMPORT or UNIMPORT-ONLY; the module dictionary remains cached as the definition source.

8.4 Recursion

User words may call themselves or other user words recursively. There is no hard-coded call-depth limit as a language semantic rule.

Native recursion-depth guard (runtime safety control, normative). A conforming implementation bounds the depth of native user-word recursion — recursive calls that must each hold a live stack frame because work remains after the call returns — with an implementation-defined depth guard (the reference implementation's guard is 256 frames). Exceeding the guard raises RecursionLimitExceeded (Section 11.1): a recoverable, diagnosable error rather than a host-level stack trap. This guard has exactly the same standing as the execution step budget of Section 5.3: it is a runtime safety control, not language semantics, its specific depth value is not conformance surface, and it does not contradict the no-hard-limit rule above or conformance item 3 of Section 16, which forbid depth limits only as language semantic rules.

Guarded tail recursion is exempt. A self-call in tail position of a COND clause body (Section 7.7.1) is executed as a backward jump that reuses the current frame, so it consumes no guard depth: a guarded tail-recursive word iterates without bound, limited only by the step budget of Section 5.3. Deep recursion that is not in guarded tail position — direct self-calls with pending work, or unguarded self-calls such as { W } 'W' DEF W — is what the depth guard bounds. Rewriting a recursion into the guarded tail form is therefore the supported way to run unbounded-depth iterations.

8.6 Word identity and content addressing

Every user word has a stable identity derived from its content, not from its name. The identity is a cryptographic hash computed over the word's normalized body together with the identities of the words it references: id = H(normalize(body) ⊕ { id(dep) : dep ∈ references }). Before hashing, every resolved reference in the body is replaced by the identity of its target, so a word's identity is independent of which names happen to be in scope when it is defined or executed. Normalization fixes a canonical form for the body — token spelling, whitespace and line breaks, the exact-rational numeric form of Section 4, and string escaping — so that semantically identical bodies always produce the same identity. A word's name, its description, and its authoring timestamp are metadata: they do not participate in the identity, and changing them does not change the identity.

Codebase and naming are separate. The set of identified definitions forms the codebase. A dictionary (Section 9.3) is a mapping from names to identities. Renaming a word is a naming-layer edit that leaves its identity unchanged. Two independently authored words with identical normalized content share a single identity automatically.

Recursion. A self-referential or mutually recursive group of words (Section 8.4) forms a reference cycle that cannot be hashed as a plain dependency tree. Such a strongly connected component is hashed as a unit, with its members referenced by position within the component, so cyclic definitions have well-defined, reproducible identities.

Editing and propagation. Editing a word produces a new identity for the new content; the previous identity is unaffected. Words that referenced the previous identity remain pinned to it until they are explicitly updated. An update re-points a dictionary's names from old identities to new ones and may be propagated to dependents on request. Adding, editing, or removing an unrelated word never changes the identity of words that did not change.

Name resolution for user words. Dependencies are stored as identities, fixed at definition time. A bare name in a stored body is a display-time label, resolved through the owning dictionary's naming map; it is never re-resolved against a global, mutable vocabulary at execution time. Consequently, adding a Core word, importing a module, or loading another user dictionary cannot recapture an existing reference. A reference to a word in another user dictionary is expressed with a qualified name (DICT@WORD) at authoring time and is likewise stored as an identity. Dependent and dependency relationships (Section 8.2) are computed from stored identities and are therefore exact and stable.

Bare names across user dictionaries. When a bare name matches entries in more than one user dictionary, resolution is decided by content identity. If every matching entry shares the same identity, they are the same word and the bare name resolves to it. If the matching entries carry distinct identities the bare name is ambiguous: it does not resolve, and a qualified name (DICT@WORD) is required to select one. Core and module resolution take precedence over this stage and are unaffected (a bare name resolves through the Core vocabulary first, then through imported modules, then through the owning dictionary, before any other user dictionary is considered).

A user dictionary is a mapping from names to identities, and import is a merge at the level of identities. Identical definitions deduplicate automatically, while differing definitions coexist as separate identities. Importing into an existing dictionary re-points the names it touches to the identities being imported; it does not destroy the prior identities or their dependents (a re-pointed name's previous identity remains in the codebase, pinned by any words that referenced it).

Sharing. Exporting a word group serializes its definitions keyed by identity together with the naming map that labels them. Importing merges definitions by identity, so identical definitions deduplicate automatically and name clashes are confined to the naming layer (Section 9.3). An imported group behaves identically in every environment, regardless of which other dictionaries or modules are present. The hash function is a defined canonical function so that independent conforming implementations agree on identities; its exact byte encoding is an implementation contract derived from this file (Section 2.1) and dictionary storage layout remains non-canonical (Section 2.3).

IMPORT loads all public Module Words from a module into the current vocabulary. IMPORT-ONLY loads only the specified Module Words.

Selectors that name Core words merely listed in a module view are no-ops for IMPORT-ONLY and UNIMPORT-ONLY, because Core words are always available and cannot be imported or unimported through a module.

Observable resolution contract of a partial import (normative). After 'M' [ 'W1' 'W2' ] IMPORT-ONLY, exactly the selected canonical Module Words resolve, and they resolve in both forms — the bare name (W1) and the qualified name (M@W1). A module word that was not selected resolves in neither form: both the bare and the qualified reference raise UnknownWord (Section 11.1). The qualified form is therefore not a backdoor around a partial import: MODULE@WORD resolution requires the word to be in the current import set, exactly as the bare name does (before any import, neither form of any module word resolves). How an implementation represents the partial-import state internally — per-word flags, a shrunk dictionary view, or otherwise — is not observable and is implementation freedom; only the resolution outcomes above are conformance surface.

9.3 Dictionary word tiers

Dictionary consists of Core Words, Module Words, and User Words. Core Words are built into Ajisai. Module Words are semi-built-in words officially provided and quality-controlled by Ajisai. User Words are words stored in user space and may be grouped by origin label.

The Core Words tier named here is narrower than the term Coreword used for contract metadata (Section 7.14): a Coreword is any built-in word with a contract entry — a Core Word or a Module Word — whereas the Core Words tier is only the permanent runtime words.

Example Words are bundled User Words provided by the reference environment. They demonstrate Ajisai features in executable form and serve as the initial template for unofficial user libraries. A user may export Example Words, modify the exported file, rename it to XXX, and import it again. The imported group is then treated as XXX Words. Because user words are identified by content (Section 8.6), an imported group is self-referential and behaves identically regardless of which other dictionaries are loaded, and identical definitions deduplicate automatically on import.

Example Words are not Core Words, not Module Words, and not a separate Dictionary tier. Module Words do not contain example, sample, or demonstration words.

9.4 SERIAL module (host-mediated serial output)

The SERIAL module exposes a serial port to Ajisai programs. Serial access is a property of the host environment, not of the runtime: the runtime never opens, writes, reads, or closes a port itself. Outbound words are effectful and produce a single host command at the IO/semantic boundary (Section 5.2); the host environment consumes that command and performs the actual port operation. The serial transport itself (a browser Web Serial implementation, a native serial backend, or none) is a host capability outside this specification. The absence of a serial-capable host is an environment condition, not a language semantic error.

A serial connection is identified by an opaque port-id text value that the host environment assigns when the user grants access. Programs treat the port id as a connection handle and thread it along the stack. The port id is Text; the runtime does not interpret its contents.

Receive model. Inbound data is delivered to a program through a per-run receive buffer (inbox). The host environment injects the bytes received on each open port before a run begins; READ drains the inbox for a port. A run therefore observes exactly the bytes that arrived since the previous run — an event-poll model, not a blocking read. Within a single run the inbox is fixed, so a run is deterministic with respect to its injected input. Each READ consumes the buffered bytes for its port; a subsequent READ in the same run with no further data projects noData (Section 11.2).

The module provides the following words:

CONFIGUREport-id baud-rate -- port-idSet the baud rate of an open port

Contract classification (Section 7.14): every SERIAL word has purity = Effectful and deterministic = false and safe_preview = false and safety_level = D. The outbound words (LIST-PORTS OPEN CONFIGURE WRITE FLUSH CLOSE) are partiality = Partial with nil_policy = RejectsNil. READ is partiality = Projecting with nil_policy = CreatesNil: it projects the no-data and disconnected conditions onto Bubble/NIL. Because they drive or observe external hardware, SERIAL words are never eligible for speculative reordering, caching, or safe_preview execution.

Misuse raises an error rather than producing Bubble/NIL (Section 11.2 "malformed use → error"): a non-text port id, a byte outside 0–255, a non-positive baud rate, or a missing operand raises StructureError or StackUnderflow. For READ, the absence of data is a well-formed outcome, not misuse: with no buffered bytes it projects Bubble/NIL with reason = noData — or reason = portDisconnected when the host has reported the port gone. The host-side outcome of a well-formed outbound command (for example a device that is physically disconnected) is reported through the host environment and does not change the runtime stack effect of the word.

The IO module (Section 9.1) provides host-mediated textual standard input and output. Like SERIAL, these words act only at the IO/semantic boundary (Section 5.2): the runtime does not perform the I/O itself; the host environment supplies the input buffer and consumes the output buffer.

10. Child Runtime

10.1 Overview

A child runtime is an isolated interpreter instance spawned from a code block. At spawn time, the child receives a snapshot of the parent dictionary. The child does not share the parent's stack or dictionary during execution. Parent and child are isolated.

10.2 Child states

StateMeaning
runningCurrently executing
completedFinished normally
failedTerminated with an error
killedTerminated by KILL
timeoutStep limit exceeded

10.3 SPAWN

Accepts a code block from the stack. Starts the child runtime. Pushes a ProcessHandle.

10.4 AWAIT

Accepts a ProcessHandle. Blocks until the child finishes. Pushes [status result-stack] where status is a string and result-stack is a vector of the child's final stack values.

10.5 STATUS

Accepts a ProcessHandle. Returns the current state as a string without blocking.

10.6 KILL

Accepts a ProcessHandle. Terminates the child runtime immediately.

10.7 MONITOR and SUPERVISE

MONITOR marks a child runtime for observation. SUPERVISE creates a SupervisorHandle that manages a group of child runtimes.

10.8 Observable guarantees and scheduling freedom (normative)

Section 10 deliberately specifies no concurrency or scheduling model. The conformance surface of the child-runtime words is exactly the following guarantees; everything else — threads, event loops, eager execution at SPAWN, deferred execution at AWAIT, preemption granularity — is implementation freedom.

  • AWAIT determinism. For a child whose block is deterministic (Section 7.14 deterministic words only), the [status result-stack] pair that AWAIT returns is a pure function of the spawned block and the dictionary snapshot: normal completion yields 'completed' with the child's final stack ({ 1 2 ADD } SPAWN AWAIT → [ 'completed' [ 3/1 ] ]), and a child that raises yields 'failed' with the child's stack as of the failure ({ NIL BOOL } SPAWN AWAIT → [ 'failed' [ NIL ] ]). A Bubble/NIL inside the child is a value, not a failure: { 0 0 DIV } SPAWN AWAIT → [ 'completed' [ NIL ] ]. Scheduling never changes an AWAIT result.
  • Parent/child isolation. As Section 10.1 states, the child receives a snapshot of the parent dictionary at SPAWN and shares neither stack nor dictionary afterwards; nothing a child does is observable in the parent except through the handle words. Definitions made by the parent after SPAWN are invisible to the child and vice versa.
  • Terminal-state stability. The states of Section 10.2 are the complete state space. Once a terminal state (completed failed killed timeout) has been reported by STATUS or AWAIT, every later observation of that child reports the same terminal state.
  • STATUS before completion is schedule-dependent. Whether, and for how long, STATUS reports running for an in-flight child depends on the scheduling model and is not conformance surface: an implementation that runs the child eagerly and synchronously at SPAWN (so running is never observed) conforms, and so does one with genuine concurrency (where STATUS immediately after SPAWN may report running). Programs must not depend on observing running.

11.2 Bubble Rule

The Bubble Rule is the user-level failure model for well-formed partial operations:

If an operation is well-formed but cannot produce a value, it produces a Bubble/NIL with a reason. If the operation is malformed, it raises an error.

In Japanese user-facing guidance this is summarized as: 「できなかった → 泡 / そもそも使い方が違う → エラー」. Internally, Bubble/NIL is represented by Value::Nil with AbsenceMetadata and a direct NilReason. A malformed operation raises an ordinary error, which propagates rather than becoming a value.

11.3 Equal-value output

Operations that produce a value equal to their input are successful. Equal-value output is not an error.

11.4 Error propagation

Ajisai has no modifier or mode that converts a raised error into a value. A malformed operation (Section 11.2) raises an error that propagates to the top level and halts the current evaluation; it is never projected onto NIL. Partial failure of a well-formed operation is handled entirely by the Bubble Rule (Section 11.2), which produces a reasoned Bubble/NIL that downstream NIL-passthrough words (Section 7.12) carry without raising, so a pipeline can end with a single VENT (^) fallback. The distinction is deliberate: "could not produce a value" becomes a bubble, while "used incorrectly" stays an error.

12. Semantic Plane

12.1 Purpose

The semantic plane holds an interpretation role for each stack position. A role is the meaning the runtime assigned to a value, not a formatting switch: rendering for both humans and AI is a pure function of (data, role). The semantic plane is separate from the data plane and does not influence computation.

12.2 Interpretation roles

Interpretation roles are applied only at explicit semantic boundaries: rendering, PRINT, and module-level output operations. A value's role and the surface style of any literal that produced it are independent: surface literal forms (Section 3.2) are convenience input syntax and are never retained as display state. Two values render identically on a given surface whenever their data and role are equal. The Stack projection \(\pi_{\mathrm{Stack}}\) and the output projection \(\pi_{\mathrm{Output}}\) are distinct surfaces, so the same value may render differently across them: a Text-role value is shown wrapped in single quotes on the Stack (so a string is distinguishable from a numeric vector) but emitted as its bare character content at the output boundary (PRINT, Section 7.9). The quotes are a Stack affordance, not data; the role itself is unchanged. Every element of a collection is rendered in its own role on each surface, so a nested Text element stays a quoted string ([ 'AB' 'CD' ]) rather than decaying to its codepoint fractions.

The ContinuedFraction role is the canonical AI-readable numeric serialization form. Machine-readable tooling that needs to round-trip exact values across the WASM boundary or diagnostic logs must request this role; the RawNumber role may lose information for lazy irrationals.

Default role of a computed scalar (normative). A scalar produced by literal input or by Coreword computation renders, on every surface and with no explicit role assignment, as follows: a rational scalar renders as its reduced numerator/denominator — the RawNumber rendering, which is why every worked example shows 3 as 3/1; a non-rational scalar (producible in the current Coreword set only via MATH@SQRT of a non-square rational and field arithmetic over such values, Section 4.2.7) renders as the nested continued-fraction form of Section 4.2.3 with its display-budget ...) truncation marker — the ContinuedFraction rendering. The "raw structural form" of the Unassigned role is, for a numeric scalar, identical to this default, so whether an implementation tags a fresh scalar Unassigned or RawNumber/ContinuedFraction is not observable and is implementation freedom. Because no current Coreword assigns the RawNumber role to a non-rational scalar, the otherwise-undefined "RawNumber surface of a non-rational scalar" is unreachable; the only compact text surface for a non-rational scalar is STR (Section 7.6.1), which is explicitly an implementation-defined, non-canonical approximation.

12.3 Observation surfaces

Section 12.1 fixes how a single value is rendered: a pure function of (data, role). The same discipline extends from one value to the whole machine. The observable state of a running Ajisai program is the tuple

\[ R \;=\; (\, B,\; S,\; O,\; V \,) \]

whose components are each defined elsewhere in this document:

  • \(B\) — the edit buffer: source text that has been entered but not yet evaluated (Section 3).
  • \(S\) — the stack as the sequence of (data, role) pairs carried by the data and semantic planes (Sections 5.2, 12.1, 12.2).
  • \(O\) — the ordered host-effect log produced by output operations and module side effects (the ordered sequence of host effects of Conformance and Identity).
  • \(V\) — the vocabulary store: the Core, Module, and User dictionaries (Sections 7, 8, 9).

Ajisai exposes \(R\) through exactly four observation surfaces, each a total, pure projection of \(R\):

\[ \pi_{\mathrm{Input}}(R)=B, \qquad \pi_{\mathrm{Stack}}(R)=\big\langle\, \rho(d_i, r_i) \,\big\rangle_i, \qquad \pi_{\mathrm{Output}}(R)=O, \qquad \pi_{\mathrm{Dict}}(R)=V \]

where \(\rho\) is the (data, role) renderer of Section 12.1 applied pointwise to each stack entry \((d_i, r_i)\). The four surfaces are named Input, Output, Stack, and Dictionary.

Two properties hold by construction and are normative:

  • Totality. Each \(\pi\) is defined on every reachable state \(R\). There is no state in which a surface has no content; an empty buffer, an empty stack, an empty effect log, or a dictionary holding only Core words are ordinary values of \(B, S, O, V\), not undefined cases.
  • Purity. Each \(\pi\) depends only on \(R\). Whenever two states have equal \(R\) their four surfaces are identical — the whole-machine generalization of Section 12.2's rule that values with equal data and role display identically.

The observation surfaces are device-independent. They define what can be observed about a running program, not how it is laid out or which surfaces are shown together; spatial arrangement, panelization, and styling remain non-observable presentation (Section 2.3). A host with no display still realizes all four projections — surfacing \(\pi_{\mathrm{Output}}\) through PRINT / IO / SERIAL (Sections 7.9, 9.4, 9.5), \(\pi_{\mathrm{Dict}}\) through LOOKUP (Section 7.8), and \(\pi_{\mathrm{Stack}}\) and \(\pi_{\mathrm{Input}}\) through host queries — rather than as on-screen areas. How the four surfaces are made simultaneously or successively visible on a given device is the subject of the Presentation Profile (Portability Profiles).

13. Fractional-Dataflow Internal Invariants

13.1 Static Mass Conservation

Ajisai treats flow mass conservation as a compile/JIT/load-time property. This is conservation of the count of values crossing a boundary; it is distinct from the conservation of their numeric total, which is the separate value-conservation invariant of Section 13.3. A Coreword Contract declares arity, consumption, production, bifurcation, and NIL-projection behavior: arity, consumption, and production are carried by the mass contract field (Section 7.14), bifurcation by the KEEP modifier (Section 13.2), and NIL-projection by nil_policy. Optimized execution paths may be entered only after those contracts have been validated for the surrounding flow.

The ordinary runtime must not maintain per-value FlowToken objects or perform step-by-step mass accounting. Flow-accounting failures such as over-consumption, unconsumed leaks, flow breaks, and bifurcation-ratio violations are contract-validation failures and must be reported by the compiler/JIT, loader, or developer diagnostics before the optimized path executes. A word whose mass is Dynamic has a data-dependent arity that the static validator does not certify; flows through such words are validated only up to that point.

13.2 Bifurcation

The ,, (keep/bifurcation) modifier retains source context while also pushing the result. Its mass relationship is specified by the relevant Coreword Contract and is statically checked with the surrounding flow. Runtime execution only performs the value-level stack effects that the contract has already proven.

13.3 Value Conservation

The mass conservation of Section 13.1 is about the count of values a word consumes and produces — a structural property checked statically before an optimized path runs. Value conservation is the orthogonal property about the numeric total those values carry: that a transform which claims to preserve a sum preserves it exactly. The two are independent — a split of [ a b ] into four parts may conserve flow-mass counts while silently violating value-mass if the parts do not add back to the inputs — so Ajisai keeps them as distinct invariants. Value conservation is the same law that double-entry bookkeeping enforces continuously (every movement debits and credits equal amounts), lifted from ledgers to ordinary dataflow.

Unlike flow-mass, value conservation is not statically inferable in general, so it is exposed as an explicit guard word, CONSERVE, rather than a compile-time check.

CanonicalSugarDescription
CONSERVE—Assert that a vector of scalar parts sums exactly to a total; pass the parts through on a proven match, else raise

Stack effect (TOP, EAT). [ total ] [ parts ] -> [ parts ], where total is a scalar and parts is a vector of scalars. When the exact sum of parts equals total, parts passes through unchanged, so CONSERVE composes transparently inside a pipeline — it is an identity on its data payload and a tripwire on its assumption. On a mismatch it raises a channel error (fail loudly); a broken conservation assumption is exactly the situation that must not flow downstream, so it is never degraded to a Bubble.

Fail loudly, never guess. CONSERVE is a guard, not an observation: a guard that cannot confirm its safe condition must not let flow pass. Over the admitted domain \(D\) (Section 4.2.7) sum-equality is exact and decidable, so for rational (money-scale) values CONSERVE either passes or raises. If the operands leave \(D\) and the equality cannot be decided within the comparison budget (Section 7.4.1), CONSERVE still raises rather than passing — only a proven equality lets flow through. This strictness is the deliberate difference between CONSERVE (a safety assertion) and EQ (an observation that may return the truth value unknown).

NIL policy. A NIL total or a NIL element of parts means an amount is absent, which cannot certify conservation (it is exactly “lost data”), so CONSERVE raises rather than passing NIL through. Its contract (Section 7.14) is Partial with nil_policy = RejectsNil, safety_level = B, and flow-mass Fixed { consumes: 2, produces: 1 }.

Relationship to QUANTIZE. QUANTIZE's residual identity \(q + r = x\) (Section 7.13) is the atomic conservation guarantee for a single rounding; CONSERVE is the aggregate one over a whole vector of parts. Together they let a program quantize an amount to a currency grid and then prove that the quantization neither invented nor lost value: quantizing the exact share 100 3 / to cents yields 3333/100 and residual 1/300, and 100 3 / [ 3333/100 1/300 ] CONSERVE passes because they reconcile to the exact total.

14. AI-first Implementation Rules

14.1 Mandatory

  • Prefer explicit, structurally searchable function and module names.
  • Keep newly created Rust source files, and files that are substantially rewritten, at or under 500 lines. Existing over-budget files are recorded as a baseline (docs/quality/file-size-baseline.json) and, as a goal, must not be changed in ways that further increase their line count.
  • Keep control flow shallow and phase-separated.
  • Separate semantic changes from structural cleanup in change management.
  • Maintain single canonical implementations; do not allow dual-mode drift.
  • Source code comments are allowed when they clarify intent, invariants, traceability, or non-obvious behavior. When source code is changed, nearby comments must be reviewed and updated so they remain accurate. Comments that merely restate obvious code should be avoided.

  • Prefer small helper extraction for duplicated control scaffolding.
  • Prefer deterministic, low-ambiguity error classification.
  • Prefer mechanically enforceable tests over narrative documentation.

15. Test Discipline

15.1 Per-Coreword contract coverage

For each Coreword, the test suite must exercise:

  • inputs that satisfy requires (success path)
  • inputs that violate requires (failure path)
  • the documented nil_policy (NIL passthrough or NIL creation, as appropriate)
  • the documented partiality (every error category in the partial case; the projection target in the projecting case)
  • effect-boundary expectations from purity (e.g. effectful words must not be reachable in safe_preview)

15.2 NIL reason coverage

Every NIL-producing path must have at least one test that asserts both the surface NIL and the structured absence metadata. Tests must verify protocol strings, not Rust Debug output.

15.3 MC/DC-style coverage for compound decisions

Words whose behavior depends on more than one independent condition (e.g. / decides on left-operand validity, right-operand validity, right-operand zero-ness, and NIL-passthrough applicability) must have tests that vary each condition independently with all others held fixed. Each condition must be shown to flip the outcome on its own.

The three-valued logic words (AND OR NOT — Section 7.5) must cover every cell of the K3 truth tables — AND and OR over the nine \(\{T, F, U\}^2\) combinations, NOT over the three \(\{T, F, U\}\) inputs — plus the NIL-interaction cases (absorbing collapse, NIL propagation, and the NIL-over-U priority rule of Section 4.5.2). The comparison words must cover at least: finite CFs always deciding to a definite truth value, equal Tier ≤ 1 irrationals — including operands composed through different histories (√8 vs √2+√2) — deciding to a definite truth value regardless of budget, and distinct irrationals deciding at a finite prefix. The logical unknown (U) outcome is confined to the reserved Tier 2 regime (Section 4.2.2) and is unreachable through the current vocabulary (Sections 4.2.7, 7.4.1).

COMPARE-WITHIN (Section 7.4.2) must cover: each decided sign (-1 0 1); Tier ≤ 1 operands — rational and algebraic, including composed equal operands — deciding regardless of budget; the Tier 2 starved case yielding unknown with diagnosis.agreedPrefix equal to the consumed budget (exercised against a type-level Tier 2 value until a word constructs one); NIL operand passthrough; and the malformed-budget / non-numeric-operand error paths.

The tiered comparison (Section 7.4.1) must cover: (i) decidability over Tier ≤ 1 at any water, including equal values built through different histories (√8 vs √2+√2) and the demotion identities of Section 7.3; (ii) the Tier 2 regime starving to unknown with a monotone, budget-bounded agreedPrefix; and (iii) the unreachability of the Tier 2 regime from the current vocabulary.

The propagation of U through comparison-dependent words (Section 7.4.3) must cover, for MIN and MAX: a decided comparison selecting the correct operand; an undecidable comparison yielding unknown with diagnosis.agreedPrefix; NIL-operand passthrough with NIL taking priority over a U-producing comparison; and the sequence-form short-circuit on the first undecidable pair. For SORT: a fully-decidable input (notably any vector of rationals) sorting to ascending order; an input with at least one undecidable pair yielding unknown for the whole result (never a partially-sorted vector) carrying diagnosis.agreedPrefix; and NIL handling. For COND: a definite-true guard firing its clause; a U guard not firing and falling through to the next clause; a U guard before a later definite-true guard selecting the later clause; and a U guard with no other match reaching the IDLE else clause, or CondExhausted when none exists.

Scalar arithmetic and comparison operate on exact representations (Section 4.2) without intermediate rounding or truncation; internal integer state is arbitrary-precision; a starved Tier 2 observation in the comparison words produces the truth value unknown (U) rather than an error or a non-deterministic answer (Section 7.4.1).
  12. Over the admitted domain \(D\) (Section 4.2.7) the six comparison relations are total and exact and never return unknown; the unknown outcome and the comparison budget are confined to COMPARE-WITHIN (Section 7.4.2) and to exact reals outside \(D\). The admitted domain is closed under ADD SUB MUL DIV, MATH@SQRT is defined on rational operands only, and leaving \(D\) (e.g. SQRT of a non-rational) is malformed use (Sections 4.2.7, 7.3).
  13. Source text contains no ( or ) outside of string literals; the nested continued-fraction form is a display/serialization artifact only (Sections 3.4, 4.2.3).

Appendix A. Gates and Water Levels (non-normative index)

This appendix is non-normative. It introduces no new rules, types, words, or protocol fields. It is a vocabulary index that maps the water metaphor used in user-facing material onto the normative mechanisms already defined above. Where this appendix and any numbered section disagree, the numbered section governs.

Ajisai does not have a global "safe mode" that wraps evaluation. Ordinary value flow is safe by design: a well-formed operation that cannot produce a value yields a Bubble/NIL (Section 11.2), an observation that cannot decide a truth value yields the logical Unknown (Stagnation; Sections 4.5.2, 7.4.1), and a malformed use raises a channel error (Section 11.1). A raised channel error propagates and halts the current evaluation; Ajisai has no modifier or mode that converts an error into a value (Section 11.4).

Two further families of controls complete the metaphor:

Gates — where flow may cross a boundary. A gate controls whether flow may cross a trust or effect boundary. Gates are not a new subsystem; they are the existing boundaries:

Portability Profiles

Core Profile

Host-independent semantics only: tokenization, vector evaluation, exact arithmetic, vectors, code blocks, the higher-order words (MAP FILTER FOLD), NIL and the logical Unknown, user definitions.

Hosted Profile

Words requiring host capabilities: NOW CSPRNG and the SERIAL module, audio, JSON export, persistence, file I/O.

Platform Profile

Concrete runtime bindings: Web/WASM, Tauri, CLI, WASI, native desktop.

Presentation Profile

The Core, Hosted, and Platform profiles classify capabilities. The Presentation Profile classifies observation: it governs how the four observation surfaces of Section 12.3 are revealed on a particular device. Like the Hosted and Platform profiles it ranks below the Core semantics — a conforming Ajisai need not present any surface visually — and a host with a display selects exactly one Presentation Profile.

A presentation profile is a labeled transition system

\[ M \;=\; (\, C,\; \Sigma,\; \rightarrow,\; c_0 \,) \]

over the surface set \(A = \{\,\mathrm{Input},\ \mathrm{Output},\ \mathrm{Stack},\ \mathrm{Dictionary}\,\}\), where:

  • a visibility configuration is the visible subset \(c \subseteq A\); the reachable configurations form \(C \subseteq \mathcal{P}(A)\);
  • \(\Sigma\) is the alphabet of abstract user operations (for example show(a), run, open-dictionary, advance, retreat);
  • \(\rightarrow \,\subseteq\, C \times \Sigma \times C\) is the transition relation;
  • \(c_0 \in C\) is the initial configuration.

The concrete \(C\), \(\Sigma\), initial configuration, and geometry are implementation freedom. The following invariants are normative; a presentation profile conforms if and only if it is a model of an LTS satisfying all of them.

  1. Partition. For every \(c \in C\), each surface is either visible or hidden — \(c\) and \(A \setminus c\) partition \(A\). A hidden surface is concealed, never destroyed.
  2. Reachability. Every surface is exposable from everywhere: for each \(a \in A\) and each \(c \in C\) there is a finite operation sequence \(w \in \Sigma^{*}\) with \(c \xrightarrow{\,w\,} c'\) and \(a \in c'\). No surface can become permanently unreachable.
  3. Non-emptiness. Every reachable configuration shows at least one surface: \(c \neq \varnothing\) for all \(c \in C\). The user is never shown nothing.
  4. Determinism. \(\rightarrow\) is a partial function: \(c \xrightarrow{\sigma} c'\) and \(c \xrightarrow{\sigma} c''\) imply \(c' = c''\).
  5. Idempotent selection. Selecting an already-visible surface is a no-op: if \(a \in c\) then \(c \xrightarrow{\mathrm{show}(a)} c\).
  6. Semantic coupling. Three constraints tie visibility to intent rather than geometry:
    • Editing is observable — any configuration in which source text can be entered has \(\mathrm{Input} \in c\).
    • Execution is observable — every run transition \(c \xrightarrow{\mathrm{run}} c'\) yields \(\mathrm{Stack} \in c'\), so the produced \((\textit{data}, \textit{role})\) stack of Section 12.3 is surfaced.
    • Selection feeds editing — any operation that inserts a selected dictionary word writes to the edit buffer \(B\), and the profile keeps the resulting \(\pi_{\mathrm{Input}}\) reachable (Invariant 2 applied to Input).

Invariants 1–6 capture the behavior that defines the Ajisai editing experience while leaving every device-specific accident free. Whether surfaces are tiled in columns or shown one at a time, the breakpoint at which a layout switches, gesture thresholds, and tap counts are tuning, not semantics — the same standing as the execution step limit of Section 5.3, which is a runtime safety control rather than a language-semantic constraint. Two hosts satisfying Invariants 1–6 present the same language even when their screens differ.

Degenerate profile. A host with no display realizes the empty presentation profile: \(M\) is trivial and no surface is ever tiled, yet the projections of Section 12.3 remain defined and are surfaced through host words instead of panels. This is why presentation is profile-grade rather than Core: the four projections are mandatory; the visibility LTS over them is host-conditional.

Conformance and Identity

An implementation is an Ajisai implementation if and only if it passes the conformance suite (tests/conformance/). The suite defines, language-independently, the correspondence between Ajisai source programs and their observable results, including the ordered sequence of host effects. This correspondence is the phenomenon of Ajisai; everything not fixed by the suite is implementation freedom.
