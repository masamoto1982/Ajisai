Introduction

Ajisai is an AI-first, vector-oriented, fractional-dataflow stack language. A program is a sequence of words: values are pushed onto a single stack, and each word consumes operands from the stack and pushes its results. Every number is an exact real — internally a continued fraction — so arithmetic never rounds.

This reference introduces each feature with short, verified examples. Every example has an "Open in Playground" button; pressing it loads the code into the app's editor so you can run it and see how it behaves.

The Reference is for using Ajisai. If you are implementing or porting the language, the single design authority is the Specification; this page is derived from it for learners and day-to-day users.

How to read the examples

The Expected value column shows the final stack exactly as the language renders it, value by value. Two display conventions matter from the start:

Numbers display as exact fractions in numerator/denominator form — the integer three renders as 3/1. Truth values display as TRUE and FALSE and UNKNOWN; the absence of a value displays as NIL.

In the text of this reference, anything with a gray background is an Ajisai token. Every symbol token defined by Ajisai is a word in disguise — + is ADD and ,, is KEEP — so the gray background is the signal that a mark belongs to the program, not to the prose.

NIL and the Bubble Rule

The failure model in one line: could not produce a value → a bubble; used incorrectly → an error. A well-formed operation that cannot produce a result projects onto NIL — a "bubble" carrying a structured reason — while malformed use (wrong types, missing operands) raises an ordinary error.

Bubbles flow: arithmetic and comparison are NIL-passthrough, so a NIL operand makes the result NIL without crashing the pipeline. At the end, VENT (^) supplies a fallback.

Sample codeExpected valueNotes
1 0 / NIL Division by zero is a bubble with reason, not a trap.

Conditionals

COND evaluates guard-and-body clauses in order and executes the body of the first guard that is definitely true. In the clause form, | separates the guard from the body and each clause occupies exactly one line; IDLE (a no-op) makes an always-true else guard.

Sample codeExpected valueNotes
[ 0 ] { [ 0 ] < | 'negative' } { [ 0 ] = | 'zero' } { IDLE | 'positive' } COND 'zero' Clause form: guard left of |, body right; one clause per line.
Open in Playground
Sample codeExpected valueNotes
[ -5 ] { [ 0 ] < } { 'negative' } { IDLE } { 'positive' } COND 'negative' Pair form: alternating guard and body blocks.
Open in Playground

A guard that evaluates to UNKNOWN does not fire its clause; evaluation falls through as if it were false. If no clause matches and there is no else clause, COND raises an error.

Managing words

DEL deletes a user word by quoted name. Dependencies are tracked: redefining or deleting a word that other words depend on requires the force modifier FORC (!). Built-in words can never be redefined or deleted. LOOKUP (?) shows the definition of a word — for a user word it returns the original source for editing.

Word names are case-insensitive (add10 and ADD10 are the same word) and conventionally follow an action-object style such as APPLY-GAIN. How a name is matched to a definition is the subject of Dictionaries and Word Identity.

Dictionaries and Word Identity

User words live in a dictionary. The words you define in the Playground go into the bundled EXAMPLE dictionary, and an imported word group forms its own dictionary named after the group. A dictionary is simply a map from names to definitions; the same name may exist in more than one dictionary at once.

Qualified names

A bare name like ADD10 is resolved for you (see below), but you can always name a word unambiguously by qualifying it with its dictionary: DICT@WORD. The qualified form bypasses resolution and points straight at one dictionary's entry.

Sample codeExpected valueNotes
{ [ 10 ] + } 'ADD10' DEF 5 EXAMPLE@ADD10 [ 15/1 ] A word you define lives in EXAMPLE, so EXAMPLE@ADD10 names the same word as the bare ADD10.
Open in Playground

Words are identified by content

Ajisai is content-addressed: a user word's identity is computed from its content — its normalized body together with the identities of the words it calls — not from its name. Renaming a word, or defining an identical body in another dictionary, produces the very same identity, while changing the body produces a new one. A word's name is a label for an identity, not the word itself.

Two consequences follow directly. Defining the same body twice does not store it twice — identical definitions share one identity and deduplicate. And because each word's dependencies are pinned to the identities they referenced at definition time, adding a core word, importing a module, or loading another dictionary later can never silently recapture an existing reference.

Resolving a bare name

A bare name is looked up in a fixed order, and the first match wins:

  1. Core words (the built-ins) — they are never shadowed by anything you define.
  2. Words from modules you have imported.
  3. The dictionary that the currently running word belongs to (so an imported group calls its own words).
  4. Any other user dictionary.

The last step is where a name can match several dictionaries at once, and here the content-addressed model decides the outcome. If every match is the same word — the matches share one content identity — the bare name resolves to it, exactly as before. If the matches are different words — they carry distinct identities — the bare name is ambiguous and does not resolve; Ajisai reports the conflict and asks you to qualify it:

For example, if a GREET with one body exists in EXAMPLE and a different GREET exists in an imported AUDIOLIB, calling the bare GREET raises Ambiguous word 'GREET': found in EXAMPLE@GREET, AUDIOLIB@GREET. Use a qualified path to specify which one you mean. Writing EXAMPLE@GREET or AUDIOLIB@GREET resolves it. Ambiguity is about a difference in content, not merely a shared name — two dictionaries holding the identical word are not in conflict.

Importing word groups

You can export a word group, edit the file, rename it, and import it again. Import is a merge by identity, never a destructive overwrite. Identical definitions deduplicate automatically, so re-importing an unchanged group is a no-op. If you edited a word and import it under a name that already exists, the name is re-pointed to the new identity while the previous identity stays in place — any words that referenced the old version remain pinned to it, so nothing breaks. The bundled EXAMPLE dictionary is treated exactly like any other in this respect; there is no special name to protect.

Because identity is content-addressed, an imported group behaves identically in every environment, regardless of which other dictionaries or modules happen to be loaded.

Output

PRINT writes the top stack value to the output area. Like any word it consumes its operand by default; with KEEP (,,) the value also stays on the stack. The Playground always shows the final stack, so PRINT is for the output channel, not for seeing results.

A string is shown on the stack wrapped in single quotes ('TEST') so you can tell it apart from a numeric vector. Those quotes belong to the Stack display, not to the value: PRINT writes the raw text, so 'TEST' prints as TEST. Quote characters that are part of the text are kept (a string whose content is T'ES'T prints as T'ES'T). Numbers and booleans print exactly as they appear on the stack.

42 ,, PRINT 42/1 The value goes to the output area and, thanks to ,,, stays on the stack.

Identity Model

Ajisai lets you write ordinary comparison words such as EQ, LT, and GTE while the language keeps several “same?” questions separate behind the scenes.

LayerUser meaningWhy it matters
Value equivalenceAre these the same mathematical value?This is what comparison words use.
Structural identityAre they stored the same way internally?Not user-observable; implementations may optimize representation.
Budgeted observationCan this equality or ordering question be decided now?If a future lazy value cannot be decided within budget, Ajisai says UNKNOWN instead of guessing.
Display equivalenceDo two rendered strings look the same?Display helps humans, but it is not the proof of equality.
Conformance equivalenceDo implementations produce the same observable results?This is how ports show they are the same Ajisai language.

For current algebraic Ajisai values, comparison is exact and total. Users do not choose an equality mode: they use the existing words, and Ajisai keeps representation details and display roles out of the value model.

Cost Model

Ajisai never rounds, yet most everyday numbers still run on a fast lane. This page is the cost model: which values are fast, what pushes a value onto a slower representation, and when a comparison can cost extra. It is practical guidance for reasoning about speed — the internal representation of a value is never observable, and nothing here changes what a program returns. For canonical behavior, the Specification is the authority (§4.2, §4.2.7, §4.3.1, §7.4).

The fast lane is small rationals

A scalar whose numerator and denominator both fit a machine integer is stored as a compact reduced fraction and runs at close to integer speed. Everything stays exact; larger or lazy values simply take a representation whose cost grows with what they hold.

ValueRepresentationCost
Small rational (numerator and denominator fit a machine integer)Compact reduced fraction — the fast lane.Fixed, near integer speed.
Big rational (needs arbitrary-precision integers)Big-integer fraction.Exact; grows with the number of digits.
Square root or other lazy irrationalLazy continued fraction.Exact; paid per continued-fraction term you actually observe.

Illustration, not a promise: on one countdown-loop benchmark the small-rational fast path ran about 1.21× faster than the general path. The cost model gives you the direction and the triggers, not a guaranteed ratio.

What moves a value off the fast lane

A value leaves the fast lane when it can no longer be a small rational. Each of these keeps the value exact — it only changes how the value is stored, and therefore how much it costs.

You do thisThe value becomes
A result grows past the machine-integer rangeA big-integer rational — still exact, cost scales with digits.
SQRT of a value that is not a perfect squareA lazy continued fraction.
Arithmetic that mixes a lazy value inA lazy result — laziness is contagious.
A vector element becomes NILNo change of representation — a dense vector marks the lane empty instead of rebuilding itself.

1000000000000 1000000000000 * 1000000000000000000000000/1 The product outgrows a machine integer, so it promotes to a big-integer rational — exact, never a float.

Vectors: dense or nested

A vector is stored one of two ways, chosen when it is built. The choice is invisible to your program — the same words, the same results — but it decides the speed of bulk numeric work.

ClassWhen you get itCost
DenseEvery element is a small rational and the shape is rectangular.Packed buffers, SIMD-friendly, fast element-wise.
NestedMixed element types, ragged rows, or big / irrational elements.Handled element by element.

To keep a numeric vector on the fast (dense) path, build it from uniform small rationals with an even shape. A single irrational or oversized element, or a ragged row, makes the whole vector nested.

When a comparison costs extra

Every comparison you can write today decides exactly — the six relations never return UNKNOWN for any value the current words can build. What changes is how much work deciding takes. Comparing two small rationals is a single integer comparison. Comparing exact irrationals (a square root, or a sum like \(\sqrt2 + \sqrt3\)) is still exact and total, but it is settled through algebraic normal-form work rather than one cheap step, so it costs more.

What you compareCost, and can it be UNKNOWN?
Small rationalsCheapest — a direct comparison. Always decides.
Exact irrationals (square roots and their sums, differences, products, quotients)Decides exactly and totally, at algebraic-normal-form cost. Never UNKNOWN.
Via COMPARE-WITHIN with a small depthYou name the partial-quotient depth; two equal lazy values can exhaust it and return UNKNOWN. This is the one window onto the budget.
Sample codeExpected valueNotes
5 3 10 COMPARE-WITHIN 1/1 Three-way compare within an explicit budget: the sign of \(a - b\).
Open in Playground
Sample codeExpected valueNotes
'math' IMPORT 2 SQRT 1 ADD 2 SQRT 1 ADD 8 COMPARE-WITHIN UNKNOWN Two equal lazy values never diverge, so a small explicit budget runs out — the only way to see UNKNOWN from today's words.
Open in Playground

The budget is measured in nearest-integer continued-fraction terms, which is the unit COMPARE-WITHIN's depth counts. The bare relations spend no observable budget on any value you can construct today; a general UNKNOWN outcome from them is reserved for exact lazy values that only future words will introduce.

Rules of thumb

  • Keep hot numeric data as small rationals to stay on the fast lane and, for vectors, on the dense path.
  • Expect square roots and other lazy values to cost more than small rationals: an observation like FLOOR pulls only as many continued-fraction terms as it needs, and a comparison settles them through exact algebraic work.
  • Big integers stay exact; if you are near the machine-integer edge, that is where the cost changes, not the meaning.
  • Reach for COMPARE-WITHIN when you want a bounded, explicit answer instead of the implicit budget.

None of this is a semantic guarantee. Representations, and the exact point at which a value leaves the fast lane, are implementation detail; the Specification (§4.2, §4.3.1, §7.4) defines what is observable.
