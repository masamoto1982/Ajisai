# Ajisai Implicit Parallelism Roadmap（暗黙並列ロードマップ・Codex向け実装指示書）

本書は、Ajisai に **暗黙的(自動)並列コンピューティング能力**を与えるための設計原理と実装フェーズを定める、Codex 向けの実装指示書である。
初期実装は Codex が担当し、最終調整は Claude が行う（§11 役割分担）。

---

## Authority（権威モデル）

- **Non-canonical.** 本書は Ajisai の意味論・ランタイム挙動・互換性方針を **定義しない**。
- **Canonical source は `SPECIFICATION.html` のみ。** 本書が意味論を制約するように見える場合は、常に仕様が勝つ。
- **実装状態の注記**: elastic/hedged 実行エンジン（hedged レース・COND ガード prefetch・fast-guarded カーネル）は opt-in の cargo feature `elastic-engine` に隔離された。デフォルトビルド（信頼コア）には含まれず、実行経路は常に greedy と同一。
- 本書が記述するのは「**どう速くするか**」であって「**何が真であるか**」ではない。観測可能な挙動を変える変更は、必ず仕様改訂を先行させること。

## 0. 目的と一文要約

> **同じソース・同じ結果・新しい概念ゼロ。違うのは速度だけ。**

並列化はランタイムが純粋性と規模から**自動で**行う。ユーザは並列のための語・注釈・スレッド数指定を一切書かない。学習曲線は上げない。割に合わない場面では静かに逐次へ退く。

設計原理を一文に畳むと:

> **机を片付け（空間）、勝ち筋が見えるまで主力を一点に集中し（8割集中）、見えたら2割の斥候に次の戦場を仕分けさせ（先行分割）、動線を交差させない（disjoint 所有）。割に合わなければ静かに逐次へ退く（Never Slower）。**

## 1. 並列性の哲学（設計原理）

本ロードマップの全判断は、次の運用規律から演繹される。これは抽象的な仕事術を計算機の語彙へ写したものである。

### 原理I — 空間は掛け算であって足し算ではない（机を片付けよ）
仕事力 = 処理能力 × 作業スペース。一軸がゼロなら全体がゼロ。どれだけコアを足しても、メモリ帯域・キャッシュが枯渇すれば出力はゼロに漸近する（memory-bound 化）。
Ajisai のアイデンティティである厳密演算（連分数・BigInt）は本質的に空間を食い、無限に伸びうる。よって「散らかしながら仕事をするな」が最優先の規律になる:
- 散らかす = アロケーション乱発・断片化・可変共有状態の書き散らし。
- 片付いた机 = SoA の `DenseTensor`（連続配置）、`Arc` による共有（コピーではなく CoW）、不変性。

### 原理II — 空間と時間は交換できる（水位予算）
空間に余裕があれば短時間で済む。空間が乏しくとも時間さえあれば終わる。最悪は両方が無い状態（スワップ＝ロッカー）。
Ajisai はこの2軸を既に「水位」として計量している:
- **時間軸** = 実行ステップ予算（`ExecutionLimitExceeded`）。
- **空間軸** = materialization / dimension の上限。
- 連分数比較で予算が尽きて `UNKNOWN` が出るのは、文字どおり「時間軸を使い切った」状態。

並列スケジューラは**新しい資源概念を足さず、この既存の水位予算の上で動く**。作業者を増やすコストは空間で支払われる、という会計を常に意識する。

### 原理III — 着手前に仕分けよ（disjoint 動線）
複数人で当たるなら、作業前に仕分ける。以降は動線が重ならず、二重作業も「誰かがやったはず」の作業漏れも出ない。計算機では:
- 着手前の仕分け = 実行前の静的解析（純粋性判定・依存 DAG・反復空間の分割）。実行中に調整しない。
- 動線が交差しない = 各ワーカーが**互いに素な領域を所有** = データ競合が原理的に起きない（不変値 + `Arc` + disjoint チャンク所有）。
- 作業漏れなし = join/barrier の正しさ。全パーティションの結果を確実に回収する。

実行時のきめ細かいロックで衝突を捌くのではなく、**先に仕分けて衝突自体をなくす**。

### 原理IV — 8割で斥候を出せ（パイプライン先行分割・過剰分散の禁止）
すべてを最初から並列化するのは、自ら多勢に無勢を作る（オーバーサブスクリプション）。
正しい運用は:
- 現タスクが**8割解消＝結果が事実上確定した点**で次へ進む。これは「勝ち筋が見えた」点であり、Ajisai では **epoch / guard 検証が通る点**（`dictionary_epoch` / `module_epoch` が安定）に対応する。
- 次へ送るのは全戦力の**約2割（斥候）**。残り8割は現ステージに張り付ける。
- **斥候の正体は「仕分け部隊」**（原理III）。斥候は次タスクを *やる* のではなく、次の戦場を *仕分ける*（次ステージの purity/DAG 分割を先行実行）。主力が到着する頃には戦場は仕分け済みで、動線が一度も交差しない。
- 計算機ではこれを **runahead / ヘルパースレッド / ソフトウェアパイプライン / decoupled access-execute** と呼ぶ。**先に並列化すべきは重い計算ではなく、軽い"仕分け"の方**である。
- 斥候の投機は**検証付き**:先行分割が残り2割の確定で覆されたら（epoch 不一致）、その斥候作業だけ破棄する。賭けは2割なので損失は小さい（hedged 検証規律のパイプライン転用）。

## 2. 不可侵の契約（破ったら全て無意味）

ユーザは並列化を信頼して丸投げする。次の3点を絶対に破らない。全フェーズの受け入れ条件に優先する。

| 契約 | 意味 | 実現機構 |
|---|---|---|
| **Same Result** | 並列実行の結果は逐次実行とビット単位で同一 | 厳密値ゆえ `assert_eq!` が成立 → 差分 proptest を CI ゲート化（§10） |
| **Never Slower** | 並列化で遅くなることが絶対にない | コスト閾値ゲート + hedged レース（逐次経路も走者に含め最悪 ≈ 逐次） |
| **Zero Syntax** | 並列のための語・注釈・スレッド数指定が一切要らない | 判断材料は `purity_table` のみ。ユーザコードに現れない |

副作用（I/O・時刻・乱数・`SERIAL@*`）の**発火順序は逐次実行と同一**でなければならない（Same Result の副作用版）。Gate 境界で直列化を保証する。

## 3. 二つの戦局（原理の適用範囲を取り違えない）

80/20 の先行分割は **すべての並列に当てはまるわけではない**。2つの戦局を分けること。

1. **タスク内（単一の巨大で均質な処理）** — 例: 100万要素への一回の `MAP`。
   戦場は均質＝すでに仕分け済み。斥候は不要で、**最初から全軍で一つの front を叩く**（rayon / SIMD で広く分割）のが正しい。→ 階層A。
2. **タスク間（異なる処理の列・グラフ）** — 例: `MAP … FILTER … FOLD …` のパイプライン。
   ここでこそ原理IV（8割で斥候）が効く。→ 階層B。

両者は競合せず入れ子になる。階層内データ並列（原理III + SIMD）と、ステージ間パイプライン（原理IV）は別レイヤーで働く。

## 4. アーキテクチャ概要（3階層）

```
階層A: データ並列   … DenseTensor 上の SIMD + rayon（MAP/FILTER/reduce/線形代数）  ← 最大ROI・タスク内
階層B: タスク並列   … パイプライン先行分割 + 子ランタイム並行 + Elastic 純粋ユニット DAG  ← タスク間
階層C: 投機並列     … 既存 hedged（テールレイテンシ削減）。スラックがあるときのみ発火
```

階層Cに関する注意: hedged は同一仕事を二戦略で競走させる＝**意図的な動線の重複**であり、原理IV の「二重作業をするな」をわざと破る。これは「作業者と空間に余りがあるときだけ許される贅沢」であり、**スラック（アイドルコア・空間予算）があるときのみ発火**する条件付き機能として実装する。

## 5. 勝利条件の再定義（何をもって「Fortran を凌ぐ」とするか）

生の f64 浮動小数点 FLOPS で Fortran に勝つことは、厳密演算を本質とする Ajisai には原理的に不可能（厳密性が必ず重い）。この土俵で競うプランは破綻する。Ajisai が実際に凌ぎうる土俵は次の3つで、ここに資源を集中する:

| 勝てる土俵 | 根拠 |
|---|---|
| 厳密有理数/整数の密テンソル並列演算 | Fortran に「自動ベクトル化される厳密有理数配列型」は存在しない。SoA `i64` レーンは SIMD 整数演算に乗る |
| 厳密性ゆえの安全な積極的再順序化 | 浮動小数点の非結合性に縛られず、ツリー並列リダクション・メモ化・投機を結果を1ビットも変えずに適用できる |
| 検証付き並列（differential parallelism） | 「並列結果 == 逐次結果」を厳密等価で機械検証できる。HPC 最大の悩み「並列化したら答えが変わった」が原理的に起きない |

目標は **「exact/verified な並列数値計算で世界最速級」**。これは達成可能かつ防御可能な主張である。

## 7. 実装フェーズ

各フェーズは独立にレビュー可能な単位とする。すべてのフェーズで §2 の3契約が受け入れ条件に優先する。

### Phase 0 — 即効: WASM を速度最適化へ
- **目的**: 最小コストで体感速度を上げる。
- **作業**: `rust/Cargo.toml` の `[package.metadata.wasm-pack.profile.release] wasm-opt = ["-Os"]` を `["-O3"]`（または検証の上 `-O4`）へ。バイナリサイズと速度のトレードオフをベンチで記録。
- **Definition of Done**: ビルド緑、Playground 起動・代表ワークロードで回帰なし、サイズ増を計測値として記録。

### Phase 1 — 基盤: Send 安全な値表現 + 差分検証の足場【最優先・他の前提】
- **目的**: 真のマルチスレッド化の前提（原理I: 机を片付ける）を整える。
- **作業**:
  1. `Rc` → `Arc` 移行。`ValueData::Vector/Tensor/Record`、`SemanticRegistry` の `Box<dyn ValueExt>` に `Send` 境界を付与。型エイリアス（例 `type VRef<T> = Arc<T>`）を一枚噛ませ段階移行。書き込みは `Arc::make_mut` で CoW。
  2. 差分検証ハーネス: 「並列カーネル結果 == 逐次カーネル結果」を厳密等価で proptest（既存 `differential_tests.rs` を拡張）。
  3. 代表ワークロードの criterion ベンチを追加し、**逐次版を恒久ベースライン**として固定（`bench-baselines/` に記録）。
- **Definition of Done**: `cargo test` 全緑、`Interpreter: Send` がコンパイル可能、差分ハーネスと逐次ベースラインが CI に存在。**この時点で機能的な速度向上はゼロでよい**。

### Phase 2 — 自動判定: 純粋性網羅 + コスト/空間ゲート【暗黙並列の頭脳】
- **目的**: ユーザが盲目的に信頼できる「自動操縦」を、保守的かつ網羅的にする。
- **作業**:
  1. `elastic/purity_table.rs` を全 Coreword + ユーザ語（`infer_purity` 伝播）で網羅。判定に迷うものは必ず `Unknown` → 逐次フォールバック（安全側）。
  2. 並列発火の唯一の門を `elastic_eligible() = pure ∧ !order_sensitive ∧ !eager`（`evaluation_unit.rs:139`）に統一。
  3. **コスト閾値ゲート**（原理II/IV）: `EvalCost` と要素数・**空間予算**から、割に合う規模でのみ並列カーネルを選択。小規模・空間逼迫時は自動で逐次。閾値はベンチで較正し定数化。
- **Definition of Done**: 全 Coreword に purity 判定が存在しテストで固定、未知語は逐次に落ちることを検証、閾値ゲートの単体テスト。
- **設計原則**: 「速くする判断」より「速くしてはいけない場面を見抜く判断」を優先する。

### Phase 3 — 本丸: DenseTensor 上の SIMD + rayon 自動カーネル【最大ROI・階層A】
- **目的**: タスク内データ並列。戦場が均質なので全軍で一点を叩く（§3 戦局1）。
- **作業**:
  1. SIMD 整数レーン演算: `is_pure_integer` かつ `Small` レーンの `MAP +/-/*`・要素比較を `wide`（安定版）等でベクトル化。`valid_mask` を SIMD マスクに対応させ NIL レーンを除外。有理数は分子・分母を別レーンで `(a/b)+(c/d)=(ad+bc)/(bd)` をベクトル化し、`gcd` 正規化を後段で実施。
  2. rayon レーン分割: `MAP/FILTER/ANY/ALL/COUNT` をコールバックが純粋なときに限り `par_chunks` で分割。`FOLD` は**結合的演算と判定できたときだけ**ツリー reduce、非結合は逐次（`order_sensitive` 厳守）。
  3. 厳密線形代数: 行列積・畳み込みをブロック化 + rayon + SIMD。看板機能。
  4. 既存 `quantized_block` / `higher_order/fast_kernels.rs` の高速カーネル層に並列カーネルを増設し、hedged の検証機構を流用。
- **Definition of Done**: 大規模整数 `MAP`・行列積が対コア数で準線形スケール、Phase 1 差分テスト全緑、Never-Slower ベンチ（小規模で逐次比劣化なし）。

### Phase 4 — 既定化: 自動並列モードをデフォルトへ
- **目的**: 「使ってさえいれば並列化される」を成立させる（Zero Syntax の完成）。
- **作業**: `elastic/execution_mode.rs` のデフォルトを自動並列モードへ昇格。`greedy` は「明示的に逐次を選ぶ」オプトアウトとして残す。WASM/CLI のモード文字列既定も更新。
- **Definition of Done**: 既定で並列が効き、`greedy` 指定で逐次に戻ることをテスト。差分テスト・Never-Slower ベンチが既定モードで全緑。

### Phase 5 — ブラウザ並列（Playground が主戦場）
- **目的**: 実用速度の主張に必須。ブラウザでも透過的に並列化。
- **作業**:
  1. `wasm-bindgen-rayon` + SharedArrayBuffer で WASM スレッドプールを有効化し、Phase 3 のカーネルをブラウザでも並列実行。
  2. COOP/COEP ヘッダ（`Cross-Origin-Opener-Policy: same-origin` / `Cross-Origin-Embedder-Policy: require-corp`）を配信に設定。GitHub Pages はヘッダ付与が困難なため `coi-serviceworker` 注入をフォールバックに用意。非対応環境は単スレッドへ自動縮退。
  3. `execution-worker-manager.ts` のスナップショット転送を、大きなテンソルについて SharedArrayBuffer バッキングへ切替えコピーを排除。
- **Definition of Done**: `navigator.hardwareConcurrency` に対し Playground ベンチが準線形スケール、非対応ブラウザで単スレッド動作を確認。

### Phase 6 — タスク並列: パイプライン先行分割 + 子ランタイム並行【階層B・原理IV】
- **目的**: タスク間の重ね合わせ。8割で斥候を出す。
- **作業**:
  1. 子ランタイムの真の並行化: `child_runtime.rs` の `run_child_to_completion` を `SPAWN` 時点で別スレッド（native: スレッド/rayon、WASM: worker）へ投入。`AWAIT` は join 待ち、`STATUS` は非ブロッキング照会。**`SPAWN`/`AWAIT` の API・意味論（仕様 §10）は不変**、中身だけ並行化。
  2. パイプライン先行分割スケジューラ: ステージ列を、現ステージが「結果確定点（epoch 安定 = 勝ち筋）」に達したら、**斥候予算（約20%）**で次ステージの purity/DAG 分割を先行実行。主力（約80%）は現ステージに張り付け、オーバーサブスクリプションを禁止。斥候の先行分割は epoch 検証付きで、覆ったら破棄。
  3. Elastic 純粋ユニットの DAG 並列発火: `EvaluationUnit.depends_on` から依存 DAG を構築し、`elastic_eligible()` なユニットのみトポロジカル順に並列スケジュール。副作用ユニットはバリアで直列化。
- **Definition of Done**: 独立な重い子ランタイム群が並行で実時間短縮、効果順序が逐次と同一、斥候の投機破棄が正しく動作。
- **スケジューラの性格**: eager fork-join ではなく **pipelined lookahead**。プログラム全体を一度に静的分割しない（それも戦力分散であり epoch 変更に脆い）。段階的に、勝ち筋が見えた段でだけ次段を仕分ける。

### Phase 7 — 検証・CI・回帰防止【全フェーズに並走】
- **作業**:
  1. 差分テスト（並列 == 逐次）を CI ゲート化。全並列カーネルに必須。
  2. Never-Slower ベンチ: `bench-baselines` 比で閾値超の劣化を CI 失敗に。
  3. 観測予算の並列健全性: comparison budget は値ごとに固定し、並列度で `UNKNOWN` 結果が揺れないことをテスト。
  4. 既存品質体系（`docs/quality/` の TRACEABILITY/VERIFICATION）へトレース追加。
- **Definition of Done**: 上記ゲートが CI に組み込まれ、緑。

### 推奨実施順
```
Phase 0 (即効)          → Phase 1 (基盤・必須) → Phase 2 (自動判定の頭脳)
→ Phase 3 (本丸・最大ROI) → Phase 4 (既定化) → Phase 5 (ブラウザ)
→ Phase 6 (タスク並列)    ／ Phase 7 は全フェーズに並走
```

## 8. 非目標（このロードマップでやらないこと）

- 並列のためのユーザ向け新語・新構文・注釈の追加（Zero Syntax 契約に反する）。
- f64 浮動小数点での生 FLOPS 競争（§5 の通り原理的に不可能）。
- BigInt（`Big`）レーンの SIMD 化（`is_pure_integer`/`Small` 高速路に限定、それ以外は逐次フォールバック）。
- 仕様（`SPECIFICATION.html`）の観測可能挙動の変更（必要なら仕様改訂を先行）。

## 9. リスクと不変条件

| リスク | 対策 |
|---|---|
| `Rc→Arc` 移行の波及が巨大 | 型エイリアス経由の段階移行 + 各ステップでテスト緑を維持 |
| 純粋性の誤判定 → 副作用の順序乱れ | 迷ったら `Unknown`→逐次。Gate 境界で副作用直列化をテスト固定 |
| 並列が逐次より遅くなる | コスト/空間ゲート + hedged レース（逐次も走者）で最悪 ≈ 逐次 |
| WASM スレッドのブラウザ要件 | service worker 注入フォールバック、非対応は単スレッド縮退 |
| 投機（斥候/hedged）の空間爆発 | スラックがあるときのみ発火。空間予算ゲートで抑制 |
| 仕様逸脱 | `SPECIFICATION.html` が唯一の規範 |

## 10. 検証・CI ゲート（要約）

1. **Same Result**: 並列 == 逐次 の厳密等価 proptest（必須・CI）。
2. **Never Slower**: ベースライン比のベンチ回帰ゲート（CI）。
3. **副作用順序**: I/O・時刻・乱数・`SERIAL@*` の発火順序が逐次と同一であるテスト。
4. **予算健全性**: comparison budget 固定で `UNKNOWN` が並列度に依存しないこと。

## 11. 役割分担と引き継ぎ手順

- **初期実装: Codex。** 本書のフェーズ単位で実装する。各フェーズの Definition of Done を満たすこと。判断に迷う設計選択（特に §2 の契約に関わるもの）は、勝手に契約を緩めず、保守的な（逐次フォールバックする）側を既定とすること。
- **最終調整: Claude。** Codex の実装に対し、(a) §2 の3契約の充足確認、(b) §1 の設計原理との整合、(c) 仕様（`SPECIFICATION.html`）逸脱の有無、(d) ベンチ/差分ゲートの妥当性、を点検し最終調整する。
- **引き継ぎの最小契約**: Codex は各フェーズ完了時に「変更ファイル一覧 / 追加テスト / ベンチ差分 / 既知の未解決点」を PR 記述に残すこと。Claude はそれを起点に最終調整を行う。
