# 能力遷移計測の設計 — 「AIレイノルズ数」議論の批判的再構成

> Status: **Non-canonical / 設計メモ.** 本書は言語意味論を一切定義しない。
> 正典は `SPECIFICATION.html` のみ。本書はエージェントベンチマークの*計測手続き*を論じる手続き文書である。
> 関連: `bench/agent-suite/protocol.md`（計測プロトコル本体）・
> `docs/quality/energy-proxy-score.md`（プロキシ命名と誠実性規律の先例）・
> `rust/src/coreword_registry.rs`（§7.14 contract metadata の実装とテスト）。

## 0. 本書の立場

ChatGPT との議論（「機械学習の創発しきい値はレイノルズ数のような支配的無次元数で
決まっているのではないか」「Ajisai は曖昧さ＝分母を下げることで、同じ能力が現れる
モデル規模のしきい値を下げられるのではないか」）を、Ajisai が既に持つ計測基盤と
誠実性規律に当てて再構成する。結論を先に置く:

1. 「AIレイノルズ数」という**名前と普遍的無次元数の主張は採らない**。
   実測の裏付けがない結果断定命名であり、`energy-proxy-score.md` が
   `energy_saved` を禁じたのと同じ理由で禁じる（§1）。
2. 採るのは反証可能な局所仮説 —— **H1:「言語・文脈側が曖昧さを下げると、
   タスク遂行に必要なモデル能力のしきい値が下がる」**。これは既存の
   `bench/agent-suite` にモデル能力階級の軸を 1 本足すだけで検証できる（§2）。
3. 「創発」観測そのものへの mirage 批判（Schaeffer et al. 2023,
   *Are Emergent Abilities of Large Language Models a Mirage?*）をガードとして
   計測設計に組み込む。二値指標だけで遷移を主張しない（§5）。

本書が定義する派生メトリクスは 2 つ: `sessionTurbulenceRatio`（§3）と
`contractCoverageRatio`（§4）。いずれも**プロキシであり、創発の予測量ではない**。

## 1. 採らないもの

| 却下対象 | 理由 |
|---|---|
| `aiReynoldsNumber` という命名 | 結果断定命名。レイノルズ数は物理的に導出された量であり、類推しか根拠のない量に借用すると測っていないものを主張する名前になる。`energyProxyScore` が `energyUsed` を名乗らないのと同じ規律。 |
| 「創発を支配する普遍無次元数」の探索 | 本リポジトリの計測範囲（1 言語 × 少数タスク × 少数モデル）では原理的に検証不能。主張可能なのは Ajisai 条件下での遷移点の移動のみ。 |
| 「Ajisai は創発しきい値を下げる」という README / SPECIFICATION 上の主張 | §2 の遷移曲線が実測されるまで書かない。実測後も「本スイートのタスク集合上で」という限定付きでのみ書く。 |

流体メタファー（層流/乱流、粘性としての診断品質）は Ajisai の水のメタファーと
整合する**説明語彙としてのみ**使用を認める。規範的な意味は一切持たない。

## 2. モデル能力階級軸（transition axis）

`protocol.md` §1 の条件 A/B に直交する第 3 の軸としてモデル能力階級を導入する。

- **T1（small）/ T2（mid）/ T3（frontier）** の 3 階級。絶対的なパラメータ数では
  なく、**同一モデルファミリー内の相対階級**として定義する（例: Haiku 級 /
  Sonnet 級 / Opus 級）。ファミリーをまたいだ階級比較は行わない。
- 各 (task, condition, tier) を独立セッションで ≥ 3 trial 実施し、
  合格率と連続量メトリクスを記録する。
- **遷移曲線** = 横軸 tier、縦軸 pass rate を条件別に描いたもの。流体の
  層流→乱流遷移曲線に対応する観測対象はこれである。
- **H1 の反証可能な形式**: 「条件 B（SKILL.md あり＝文脈側の曖昧さ低減）の
  遷移は、条件 A より低い階級で起こる」。より強い将来形は、他言語ベースラインを
  条件に加えた「Ajisai 条件の遷移は他言語条件より低い階級で起こる」。
- 遷移が観測されない・逆方向に動く結果も `protocol.md` §5（負けの公開）に従い
  そのまま公開する。

## 3. `sessionTurbulenceRatio` — セッション乱流度

エージェントの write → run → fix ループがどれだけ「乱れた」かを表す無次元比。

```
draftTokensOut         = 最初に提出した解に至るまでの completion トークン数
repairTokensOut        = tokensOut − draftTokensOut （最初の提出以降の修正分）
sessionTurbulenceRatio = repairTokensOut / draftTokensOut
```

- `onePass` のセッションは定義上 0（層流）。修正往復が多いほど大きい（乱流）。
- トークン ÷ トークンなので無次元であり、モデル・タスクをまたいだ比較に
  絶対量より頑健である。ただし**比較はメトリクス版数（§6）が同じ記録同士に限る**。
- `tokensIn` を分母に**含めない**理由: 条件 A/B は提供資料の量が意図的に違うため、
  prompt トークンを混ぜると条件差そのものがノイズとして混入する。
- 診断品質（`errorQuality`、構造化された `why`、NIL の理由メタデータ）が高いほど
  少ない往復で収束する —— メタファー上は「流れを層流に保つ粘性」に相当する ——
  という因果は**仮説であって定義の一部ではない**。乱流度はあくまで観測値である。

### 記録方法

`draftTokensOut` はセッションログから最初の解提出時点で切って数える。切断点は
「エージェントが最初の完全な解ファイルを書き終えた時点」とし、results ファイルの
Notes にどう切ったかを一行残す。切断点が復元できないセッションでは両欄を空欄に
する（`protocol.md` §5: 値をでっち上げない）。

## 4. `contractCoverageRatio` — contract 被覆率

H1 の「分母（曖昧さ）」の代理値。最終解が、機械可読な契約で覆われた語表面に
どれだけ乗っているかを測る。

```
contractCoverageRatio = 完備な §7.14 contract metadata を持つ定義に解決される語出現数
                        ────────────────────────────────────────────
                        最終解の全語出現数（literal・修飾子・構文記号を除く）
```

- 「完備」とは `purity` / `partiality` / `nil_policy` / `safety_level` /
  `mass`（§13.1）がすべて宣言されていること。
- **Coreword は既に 100% 被覆**であり、
  `aq_ver_contract_a_every_word_has_contract_metadata`
  （`rust/src/coreword_registry.rs`）がそれをテストで固定している。
- したがってこの比を動かすのは:
  - **モジュール語** — 既定値＋ `contract_override`
    （`rust/src/interpreter/modules/module_builtins.rs`）で契約が付与される。
    付与漏れがあれば分母に残る。
  - **ユーザー語** — 現状 contract 宣言機構がないため、当面すべて
    **uncovered として数える**。将来ユーザー語に契約宣言が入れば分子に移る。
    この扱いにより「解答がユーザー語定義に逃げるほど被覆率が下がる」という
    保守的な性質になる（sparse counter が deduction を得ないのと同じ向きの保守性）。
- 分母から除くもの: 数値・文字列・vector リテラル、modifier（`TOP`/`STAK`/
  `EAT`/`KEEP`）、`^` などの構文的記号、コメント。これらは「語の契約」という
  概念の適用外であるため。
- 計測は当面、最終解ファイルに対する手動またはスクリプト集計でよい。将来は
  `ajisai coverage <file>` のようなサブコマンドで機械化するのが自然だが、
  **その実装は本書のスコープ外**である（本書は定義のみを固定する）。

### この比が主張できること・できないこと

- 主張できる: 「この解答は語出現の X% が契約完備な表面の上にある」。
  条件間・タスク間での被覆率と `fixCount` / 乱流度の相関の観測。
- 主張できない: 被覆率が高いから曖昧さが低い、という因果。被覆率は
  H1 の分母の**代理値候補**であり、妥当性自体が検証対象である。

## 5. mirage ガード（二値指標だけで遷移を主張しない）

Schaeffer et al. 2023 は、創発的ジャンプの多くが「正解/不正解の二値でしか
測らない指標が作る見かけ上の不連続」であり、連続指標で測り直すと滑らかな
スケーリングに戻ると論じた。本計測はこれを次の規律で組み込む:

1. 遷移曲線（pass rate = 二値集計）を提示するときは、**必ず同じデータの連続量**
   （`fixCount`・`sessionTurbulenceRatio`・`tokensOut`）の tier 推移を併記する。
2. 遷移が二値でのみ現れ、連続量が tier に対して滑らかな場合、それを
   「指標アーティファクトの可能性が高い」と結果ファイルに明記する。
3. 二値と連続量の両方が同じ tier 付近で急変する場合に限り、「遷移らしき挙動」
   と記述してよい。それでも「創発」という語は結果ファイルでは使わない。

## 6. バージョニングと誠実性

`docs/quality/energy-proxy-score.md` の規律をそのまま踏襲する。

- **`TRANSITION_METRICS_VERSION = 1`**。§3・§4 の定義（切断点規則、分母の
  除外規則を含む）への**いかなる変更も**この版数を増分し、本書を同一変更内で
  更新しなければならない。版数が異なる記録同士は比較不能である。
- 名前は観測記述的に保つ（`sessionTurbulenceRatio` / `contractCoverageRatio`）。
  `emergence*` / `intelligence*` / `aiReynolds*` のような結果断定名は使わない。
- 実測が揃うまで、README / SPECIFICATION はこれらのメトリクスに基づく能力主張を
  一切書かない。書けるのは「Ajisai は遷移計測の手続きと代理指標を定義し、
  結果を負けも含めて公開する」という手続き上の事実のみ。

## 7. 反証条件（この設計が失敗と判定される条件）

- 条件 B の遷移階級が条件 A より下がらない、または乱流度が下がらない
  → H1 の「文脈側の曖昧さ低減」部分は本スイート上で支持されない。結果は
  公開したまま残す。
- `contractCoverageRatio` が `fixCount` / 乱流度と無相関
  → 被覆率は分母の代理値として不適。§4 を廃止するか再定義し、版数を増分する。
- 二値でのみ遷移が見え連続量が滑らか → mirage の可能性を明記し、
  遷移主張を取り下げる。

いずれの場合も、失敗した仮説の記録を削除しないこと（`protocol.md` §5）。
