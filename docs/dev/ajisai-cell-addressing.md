# Ajisai セルアドレッシング設計ノート — 座標をやめ、名前で指す

Status: design note (non-canonical, 2026-07-04)

## 位置づけ

- 本書は **非正典**。新エンジンの正典は `SPECIFICATION-NEXT.html` ドラフト。
- 本書はセルの参照方式（アドレッシング）の再設計を **提案** する。承認後、
  正典ドラフトの §3.5 / §5.3 / §5.2 を書き換える。現行ドラフトは A1 座標を
  canonical な参照としているため、本提案はそこへの変更提案である。

## 1. 問題 — なぜ A1（英字×数字）では不十分か

現行ドラフト §5.3 は正直に自分の弱点を書いている:

> Cell addresses use the A1 convention … Addresses are **positional** and refer to
> the grid slot, not to a value identity. Structural edits … **rewrite** affected
> references in formulas.

A1 方式は三つの欠陥を抱える。新型 Ajisai では**セル結合が当たり前**なので、
一つ目が致命的になる。

1. **結合が格子の前提を壊す。** A1 は「すべての (列, 行) が別々のセルである
   一様格子」を前提にする。`B2:C3` を結合すると、被覆座標 `C2` `B3` `C3` は
   「穴」になる。廃案プラン §3.4 は「被覆アドレスは anchor に解決する」という
   *パッチ*で凌いだが、これは「座標が唯一の名前」という誤った前提を温存したまま
   例外処理を足しているだけで、参照の静かな誤りの温床になる。
2. **アドレスが位置であって同一性でない。** 行を 1 本挿入すると下の全アドレスが
   ずれ、数式を書き換えねばならない（§5.3 が明記）。純粋関数型・再計算モデルに
   とって「参照先が編集で勝手に変わる」のは異物である。
3. **座標と名前を混同している。** A1 は「格子上のどこか（座標）」と「数式が
   どう指すか（名前）」を一つの記号に押し込む。セルに安定した同一性がないので、
   座標を名前の代用にせざるを得ない — これが 1・2 の根本原因。

## 2. 原理 — 幾何と同一性を分離する

新型 Ajisai は既にあらゆる層でこの分離を貫いている:

| 層 | 正典（同一性） | 表示（幾何・見た目） |
| --- | --- | --- |
| 数式 | 連接 canonical 形 | infix 表面形 |
| 数値 | 厳密実数 | 表示丸め（QUANTIZE） |
| 文書 | テーブル＋辞書 | シート・キャンバス配置・書式・結合 |

アドレッシングだけがこの原則の外にいる。**A1 座標は「幾何」の側の概念であり、
「同一性」の側に置いてはならない。** 是正はこうなる:

> **セルは名前で指す。座標は表示・入力の補助にすぎない。**
> （The A1 coordinate is presentation; the name is canonical.）

これは他の層と同じ「canonical / surface」分離をアドレッシングにも適用するだけで、
新しい原理を導入しない。器のメタファーにも合う — 器には**ラベル（名前）**を貼る
のであって、棚の座標で呼ぶのではない。

## 3. 提案 — 名前空間つき名前参照（nominal addressing）

### 3.1 参照は名前で、名前空間は入れ子

- テーブル名・シート名は **名前空間** として機能する（ユーザー要望）。
  テーブル名は文書内一意（§5.2 既定）なので、シート階層は表示上のグループに
  すぎず、参照パスには不要。参照は原則 `テーブル.メンバー` の 2 段で足りる。
- 参照の canonical 形（連接）と infix 表面形の対応:

  | 指すもの | 連接 canonical | infix 表面 |
  | --- | --- | --- |
  | 同じ行の列値（列式内） | `@価格` | `[@価格]` |
  | 列全体（範囲） | `売上.価格` | `売上[価格]` |
  | 名前つきセル（landmark） | `合計` / `請求書.合計` | `合計` |
  | 関数表の適用 | `3 4 三角形.面積` | `三角形.面積(3, 4)` |

  現行の `@` 修飾（`TABLE@…`）は `.`（ドット）名前空間解決に一般化する
  （`.` は表計算ユーザーに構造化参照・メソッドとして馴染み深い）。

### 3.2 三種類の参照可能なものだけを持つ

新型 Ajisai は A1 の「任意の座標を指せる」能力を**意図的に捨てる**。指せるのは:

1. **列**（`テーブル.列`）— 集約語（SUM 等）の範囲引数。§5.4 の範囲＝参照に一致。
2. **同じ行の列値**（`@列`）— 列式（§5.5）の行文脈。横方向の参照はこれで尽きる。
3. **名前つきセル**（landmark / 定義名）— 帳票の合計・入力欄など、個別に指したい
   スカラーのセル。参照されないデータセルは名前を持たなくてよい。

「A1 で任意の他行セルを指す」という**壊れやすい斜め参照は設計から消える**。
行はレコードであり、特定行の参照は**キーによる LOOKUP**（後続フェーズの語）で
行う（序数でなく意味のあるキーで指す）。これは §5.4（範囲＝参照）・§5.5（列式）と
既に整合しており、A1 斜め参照はもともとほぼ不要になっている。

### 3.3 これで結合が「問題」でなくなる

結合の被覆座標問題（§1-1）は、**座標を同一性から外した瞬間に消滅する**:

- 結合領域は**一つの器**である。名前を持てば `請求書.表題` のように指せる。
- 被覆座標という概念自体が無い（座標が同一性でないので「解決先」も要らない）。
- 幾何情報（原点・rowspan・colspan・キャンバス位置）は表示メタデータ（§13.2）。
  結合＝「一つの名前つき器が矩形を占める」だけ。廃案プラン §3.4 の
  「anchor 解決」パッチは丸ごと不要になる。

帳票（神Excel）はこの方式と特に相性がよい。帳票とは「ラベル付きフィールドの
レイアウト」であり、各フィールドを名前で指す（`請求書.顧客名`）のは、
「C7 のセル」で指すより本来的に正しい。結合された見出しは幅広の器で、名前は
必要なときだけ付ける。

### 3.4 A1 座標は GUI の補助として残す（親しみやすさの保持）

canonical から外すが、捨てはしない。表計算ユーザーの親しみを GUI 層で保つ:

- 名前ボックス（§5.3 / §13）は `C4` の入力で**その位置へジャンプ**できる（ナビ）。
- セルをクリック → その場に名前が無ければ、数式バーは一時的な座標参照を
  提示しつつ「名前を付けますか？」を促す（landmark 化への導線）。
- ただし**保存される数式・正典参照は名前**。座標は infix 表面形と同じ「表面」で
  あり、意味論は名前側にのみ置く（§3.1 と同じ canonical/surface 分離）。

## 4. 検討した代替案と却下理由

| 案 | 内容 | 評価 |
| --- | --- | --- |
| **A. A1 継続 + anchor パッチ**（廃案プラン §3.4） | 座標を canonical のまま、結合被覆を anchor に解決 | §1 の三欠陥を温存。結合ごとに例外規則。却下 |
| **B. 不変セル ID + 名前** | 各セルに不可視の安定 ID を振り、名前は ID の別名 | 同一性は完璧だが、ID は人にも AI にも読めず、帳票の手書き感を損なう。ID は実装詳細として内部に持ってよいが、**参照面には名前を出す**のが本提案 |
| **C. 関係代数 (table, column, row-key)** | セル＝(表, 列, 行キー)の三つ組 | データテーブルには最適で本提案の列参照はこれの射影。ただし帳票の自由レイアウトを (行キー) で表せない。**表向きの部分は本提案が採用、帳票は名前つきセルで補完** |
| **D. 本提案（名前空間つき名前参照）** | 幾何と同一性を分離し、名前で指す。座標は GUI 補助 | 既存の canonical/surface 原則の再利用。結合が自然消滅。採用 |

## 5. 正典への反映（承認後）

1. §5.3 を全面改稿: 「A1 は位置的アドレスで構造編集が参照を書き換える」→
   「参照は名前。座標は非正典の GUI 補助」。構造編集で参照を書き換える規定
   （fragile な §5.3 現行）は、名前参照では**そもそも書き換え不要**になるので削除。
2. §3.5: 参照文法を `@列` / `テーブル.列` / 名前つきセル / `テーブル.列`適用 に改稿。
   A1 文法と「裸トークン＝セル参照」の閉じた語彙規定を、名前解決規定に置換。
3. §5.2: 列名・名前つきセル名の一意性と名前空間解決を規定。
4. §5.6 / §6.6（関数表）: `TABLE@COL` を `テーブル.列` に表記統一。
5. §13.2: 結合＝表示スパン、正方形セル標準（別途の UI 指示）、A1 ナビ補助を明記。

## 6. 未決定事項（承認前に判断を仰ぐ）

1. **区切り記号**: 名前空間の区切りを `.`（`売上.価格`）にするか、現行の `@`
   （`売上@価格`）を残すか。`.` は構造化参照に馴染むが、`@` は既に実装済み。
2. **名前つきセルの宣言**: どう名前を付けるか（セル選択→命名 UI、ヘッダ規約、
   専用の定義名リスト）。
3. **行キー参照 LOOKUP** の語surface と導入フェーズ（§5.4/§5.5 の範囲・列式で
   当面代替できるので後続でよいか）。
4. **A1 座標を完全に GUI 限定にするか**、CSV 取り込み等の互換のために限定的な
   座標入力を残すか。
5. 既存の「裸 `A1` 形トークンはセル参照」規定（§3.5 現行）の撤廃可否 —
   撤廃すると `A1` はただのテキスト/名前になり、表計算の刷り込みと衝突しないか。
