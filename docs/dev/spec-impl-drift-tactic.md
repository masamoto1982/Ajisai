# 仕様・実装乖離の抑制戦術 — タイムスタンプ裁定の批判的再構成

> Status: **Non-canonical / 設計メモ（§2.2）.** 本書は言語意味論を一切定義しない。
> 正典は `SPECIFICATION.html` のみ。本書は乖離抑制の*運用戦術*を論じる手続き文書である。
> 関連: `SPECIFICATION.html` §2.4（四層ドキュメントモデル）・§2.5（権威順位）・§16.1（第二権威の禁止）・
> "Conformance and Identity"／`tools/ajisai-repro/DIVERGENCE-ANALYSIS.md`（A/B/C 乖離類型の実証）。

## 0. 本書の立場

ChatGPT との議論（数式→仕様→Reference→conformance→実装の一方向導出、および
タイムスタンプ裁定）を、**Ajisai が既に持っている権威構造と実証済みの乖離分布**に
当てて再構成する。結論を先に置く:

1. ChatGPT 案の「**数式を最上位の権威にする**」は**採らない**。Ajisai が §2.4/§2.5/§16.1 で
   意図的に拒否済みの構造であり、Conformance Checklist 項目1（第二権威の禁止）に違反する。
2. ChatGPT 案の「四層モデル」は **§2.4 として既に存在する**。新規性はない。
3. ChatGPT 案の「**タイムスタンプ裁定**」は、Ajisai の実際の乖離分布に対して**最も稀な
   乖離クラスだけを最適化し、支配的なクラスに無力**である。タイムスタンプは捨て、
   **「観測可能挙動はすべて正典アンカーへ追跡可能でなければならない」+「同点はスイートが裁く」**
   に差し替える。これが本書の提案する戦術である。

## 1. なぜ「数式を最上位」を採らないか

ChatGPT は裁定順序を `数式 ▷ 仕様書 ▷ conformance ▷ … ` とし「最上位の権威は数式定義」と述べる。
Ajisai の §2.4 表はこう定める:

| 層 | 権威 |
|---|---|
| Mathematical formalization | **Descriptive**（本文を制約・動機づけるが決して上書きしない） |
| Specification（本文） | **Canonical** |
| Reference | Derived |
| Authoring discipline | Non-canonical |

数式を canonical にすると:

- **二重正典化**になり §16.1（"It does not introduce a second design authority"）違反。
  これは Conformance Checklist の**第一項目**であり、最も重い不変条件。
- Ajisai の数式（`docs/dev/ajisai-mathematical-formalization.md`）は自身を
  「仕様に対する第二の権威ではなく、仕様が定める現象を記述するモデル」と明言している。
  数式を上位に置くと、この自己規定と矛盾する。
- 実務上も危険: 数式の方が美しく書けてしまうため、実装の制約（予算付き比較の U、
  完全平方の有理縮退、GET の非消費など §9.x 所見群）を数式が*取りこぼした*とき、
  数式が実装を「間違い」と断罪してしまう。実際にはそれらは仕様化すべき設計判断である。

**採用方針**: §2.5 の権威順位（本文 ▷ 数式 ▷ conformance/law tests ▷ 生成データ ▷ Reference ▷
style ▷ 実装）を**そのまま維持する**。数式は「乖離を即座に暴くオラクル」(§9.3) として使うが、
裁定者には**しない**。

## 2. なぜ「タイムスタンプ裁定」が Ajisai に効かないか

ChatGPT 自身が裁定を「新しい方が勝つ」から「乖離検出の暫定ヒント」へ後退させた点は正しい。
だが Ajisai に当てると、タイムスタンプは**ChatGPT が見積もった以上に無力**である。理由は二つ。

### 2.1 「誰が勝つか」は §2.5 で恒久的に決着済み

タイムスタンプは「どちらが新しい設計意図か」を推定する道具だが、Ajisai では
**本文が常に勝つ**ことが §2.1/§2.5 で確定している。裁定の出力は最初から自明
（本文 ▷ 実装）であって、タイムスタンプが追加する情報は無い。タイムスタンプ裁定は
「権威がフラットな言語」向けの道具であり、単一正典を持つ Ajisai には不要な機構。

### 2.2 支配的な乖離クラスは「タイムスタンプを付けられない穴」

`DIVERGENCE-ANALYSIS.md` の実証（79本中15本相違）が示した分布:

| 類型 | 件数 | 内容 | タイムスタンプは効くか |
|---|---|---|---|
| **A 類（仕様の穴）** | 9 | 実装＋スイートは一貫設計を持つが、**本文が署名・規則を書いていない** | **効かない**。存在しない項目に決定日時は付けられない |
| **B 類（実装が明文違反）** | 2 | 本文に明確な規定、実装が外れ、スイート沈黙 | 一見効くが、実体はスイートの被覆漏れ |
| **C 類（本文内の不整合）** | 3 | 本文の複数箇所が食い違う | 本文内なので「仕様 vs 実装」軸でない |

支配的な A 類（9/15 = 60%）は**仕様の不在**である。不在の項目に決定タイムスタンプは
存在し得ない。タイムスタンプ裁定が想定する「仕様項目 X(t1) と 実装挙動 Y(t2) の衝突」は、
Ajisai では**最も稀**（B 類 2件）にしか起きない。すなわちタイムスタンプは、
**起きていない問題のために台帳を増やし、起きている問題（穴）を検出できない**。

### 2.3 タイムスタンプは脆く、改竄に弱い

仮に手書きの決定タイムスタンプを各規範項目に付けても:

- 整形・節番号の振り直し・文言の言い換えで容易に「更新」され、意味の変化と区別できない。
- 「いつ意図的に変えたか」は、**git が既に正確・不変・改竄困難に記録している**。
  手書きタイムスタンプは git 履歴の劣化コピーにすぎない。

## 3. 提案戦術 — 追跡可能性ゲート + スイート裁定

タイムスタンプ裁定が解こうとした問題（乖離を制度的に発見可能にする）を、Ajisai の
実態に合う形へ差し替える。**新しい正典機構は導入しない**（既存インフラの結線のみ）。

### 3.1 原理: 「観測可能挙動 → 正典アンカー」の全射を不変条件にする

A 類（穴）を機械的に捕まえる唯一の方法は、タイムスタンプではなく**被覆の全射性**である:

> **追跡可能性ゲート**: conformance suite が pin する各観測可能挙動、および各 Coreword 契約
> （§7.14）は、`SPECIFICATION.html` の**安定アンカー（規範項目 ID）**へ少なくとも一つ
> 追跡可能でなければならない。アンカー無しのスイートケース＝**検出された A 類の穴**。

これは既存の三資産を結線するだけで実現する:

- `docs/quality/TRACEABILITY_MATRIX.md`（要件 ID → 実装 → 検証）
- `docs/formalization-coverage.json`（正準語 → 代数族・導出辺・primitive 登録）
- `docs/primitive-test-map.json` / `tests/conformance/`（ケース ↔ 挙動）

足りないのは一本の辺だけ: **conformance ケース → 正典アンカー**。これを必須化し、
`scripts/check-formalization-coverage.mjs` と同型のチェッカ（`scripts/check-spec-anchor-coverage.mjs`）を
追加し、CI ゲートにする。アンカーを持たないケースが現れたら CI が落ち、A 類の穴がその場で可視化される。

### 3.2 provenance はタイムスタンプではなく「アンカー上の git」で取る

「いつ・どちらが新しい意図か」が必要な場面（主に C 類の本文内不整合の調停）では、
手書きタイムスタンプではなく**安定アンカー上の git 履歴**を使う:

- 各規範項目に**安定 ID**を与える（本文 HTML は既に `id=` アンカーを持つ。これを
  節レベルから**規則・語契約レベル**へ細粒度化する。例 `id="rule-mod-by-zero"`、
  `id="contract-GET"`）。
- 「この規則の規範内容が最後に意図的に変わったのはいつか」は
  `git log -L` / `git blame` を当該アンカー断片に当てて得る。これは不変・精密・改竄困難で、
  手書きタイムスタンプの上位互換。
- 実装側の「いつ導入・変更したか」も同様に当該実装行の git blame で得る。

すなわち ChatGPT の「仕様タイムスタンプ／実装タイムスタンプ」は、**安定 ID + git** に
還元すると、台帳を増やさずに同じ（むしろ強い）情報が得られる。

### 3.3 裁定者はクロックではなくスイート

乖離の**方向**（impl→spec か spec→impl か）は時計に決めさせない。`DIVERGENCE-ANALYSIS.md`
が確立した規律をそのまま規則化する:

> **スイート裁定規則**: 観測挙動と本文が食い違ったとき、
> - その挙動を **conformance suite が pin している** → 意図的な設計判断。**spec→impl**
>   （本文を加筆して挙動を正典化）。スイートは "Conformance and Identity" により
>   *実装間同一性の運用的定義 (L5)* を担うので、pin 済み挙動は設計の表明とみなす。
> - **スイートが沈黙している** → 実装バグの候補。law test / 数式オラクル(§9.3)に照らし、
>   明文違反なら **impl→spec**（実装を本文へ合わせる）。
> - **本文内が食い違う**（C 類） → §2.5 と git provenance を補助に設計者が決定（要判断）。

時計ではなく**スイート（＝同一性の定義そのもの）**が同点を裁く。これが Ajisai 固有の
正しい裁定基盤である。

### 3.4 変更プロセスの相互拘束（ChatGPT が正しく捉えた部分の維持）

ChatGPT の「仕様変更は必ずテスト変更を伴う」という相互拘束は正しい。Ajisai 流に固定する:

- **本文の規範項目を加筆／改訂** → 同一 PR で conformance ケースか law test を追加し、
  追跡可能性ゲート（§3.1）を満たす。
- **Coreword の追加／改名** → §7.14 契約 + 追跡アンカー + テストを同時に
  （Conformance Checklist 項目7・8・10 で既に半ば強制済み）。
- **数式の改訂** → Descriptive のまま。本文を*動機づける*が、本文・テストを伴わない数式変更は
  正典に影響しない（§2.4）。

## 4. ChatGPT 案との対応表

| ChatGPT 提案 | Ajisai での扱い |
|---|---|
| 数式を最上位の権威に | **却下**。§16.1 違反。数式は Descriptive のまま（§2.4） |
| 四層（数式/仕様/Reference/conformance） | **既存**（§2.4）。矢印は逆（本文が canonical） |
| 一方向導出 数式→仕様→Ref→test→実装 | 部分採用。導出ではなく**相互拘束**（§3.4）として運用 |
| 仕様項目に決定タイムスタンプ | **却下→差替**。安定アンカー + git provenance（§3.2） |
| 実装にタイムスタンプ | **却下→差替**。実装行の git blame（§3.2） |
| 新しい方を暫定的意図として扱う | **却下**。§2.5 で本文が恒久的に勝つ。方向はスイートが裁く（§3.3） |
| 乖離を制度的に発見可能に | **採用・強化**。追跡可能性ゲートを CI 必須化（§3.1） |

## 5. 最小実装ステップ（提案）

1. `SPECIFICATION.html` の規範項目に細粒度の安定 `id` アンカーを付す（語契約・各規則）。
   表示には影響しない純粋な機械可読アンカー。
2. `tests/conformance/index.html` の各ケースに `data-spec-anchor`（複数可）を付す。
3. `scripts/check-spec-anchor-coverage.mjs` を追加: 全ケースがアンカーを持ち、
   アンカーが本文に実在することを検査。アンカー無し＝A 類の穴として **CI fail**。
4. §2.5 の直後に、**スイート裁定規則（§3.3）**を一段落で明文化する（タイムスタンプは導入しない）。
   ※これは正典本文の改訂なので、別 PR・設計者承認のうえで行う。本書はその提案に留める。
5. `git`-based provenance の運用メモを `docs/quality/TRACEABILITY_POLICY.md` に追記。

## 6. 一行サマリ

> §2.5 の単一正典・権威順位を維持し、「新しいタイムスタンプが勝つ」を捨てて
> **「観測可能挙動はすべて正典アンカーへ追跡可能」+「同点は conformance suite が裁く」**
> に差し替える。provenance は手書き日時ではなく**安定 ID 上の git** で取る。
> 追跡不能な挙動・アンカー無しのスイートケースを **CI が落とす**条件にすることで、
> 支配的な乖離クラス（仕様の穴）を制度的に発見可能にする。
