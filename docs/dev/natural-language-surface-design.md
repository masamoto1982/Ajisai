## 1. 位置づけ：Ajisai は「自然言語の着地点」である

Ajisai を**人間が手書きして楽な言語**にする方向は採らない。連分数・スタック・修飾子・
水流モデルを生身で扱う負担は、一般的な高級言語より大きい。Ajisai の強みは「手書きが
楽」ではなく「**意味の検証可能性が高い**」ことにある。

したがって自然言語プログラミングの形は「自然言語を直接実行する」ではなく、次の
**翻訳ループ**である。Ajisai は、自然言語の曖昧な要求が落ちる先の**検証可能な中間表現**
として機能する。

```
自然言語の要求
   │  ［案2 軽量版］契約照合とフロー質量保存で「壊れた計画」を実行前に弾く
   │  ［案3］操作意図を修飾子（TOP/STAK × EAT/KEEP, VENT）の有限格子へ分類する
   ▼
Ajisai 計画（契約検査済み）
   │  ［案4］計画時に決めきれない → UNKNOWN として逆質問（推測で潰さない）
   ▼
実行
   │  ［案4］実行時に観測予算切れ → UNKNOWN + agreedPrefix で逆質問
   ▼
   │  ［案1］DebugDiagnosis / AiDiagnosticPayload を自然言語へ射影して説明
ユーザーへの説明・次の一手
```

4 案は横並びの独立機能ではない。**案2が動力源**であり、案3はその語に付く操作様態、
案4は案2の出力の不確定を捌く対話、案1はループ両端（前向き翻訳と実行結果）の説明である。

## 2. 最上位の設計制約：学習曲線を上げない（段階開示）

**本層の成功条件は「ユーザーが仕組みを知らなくても便利に使える」ことである。**
TOP/STAK/EAT/KEEP、契約メタデータ、NIL と UNKNOWN の区別、連分数 —— これらを
ユーザーが理解していなくても恩恵を受けられねばならない。機構は既定では隠れ、
ユーザーがより細かい制御を望んだとき、または逆質問が真に必要なときにだけ表面化する。

`three-layer-documentation-model.md` の段階開示を、対話サーフェスへ拡張する。

| レベル | ユーザーの状態 | 提示 | 機構の露出 |
|---|---|---|---|
| **L0 既定** | 「やりたいことを書いた」 | 平易な言葉で入出力。専門語なし | ゼロ |
| **L1 明確化** | 「意図が一意に決まらなかった」 | 平易な言葉で逆質問（案4） | ゼロ（用語を出さない） |
| **L2 熟練** | 「中を見たい／細かく制御したい」 | 生成された Ajisai コード、構造化診断、修飾子の選択 | 全開 |

設計判断はすべてこの制約から導かれる。

- **案3** は既定値が賢くあること。`EAT` と `TOP` を既定とし、ユーザーが操作様態を
  意識せずに済むようにする。修飾子の選択を表面化するのは、案4が真の曖昧さを検出した
  ときだけ。L0 では「先頭か全体か」「残すか消費するか」を一切問わない。
- **案4** の逆質問は L1 として平易な言葉で行う。「`EAT` ですか `KEEP` ですか」ではなく
  「元の配列も残しますか？」。`UNKNOWN`・`agreedPrefix` は内部表現であり、文面に出さない。
- **案1** の説明は既定で機構を含まない平文。`recoverability` を行動文に射影する（§3）。
  「詳しく」を選んだユーザーにのみ L2 として構造化診断と生成コードを開示する。
- **案2 軽量版** の契約検査は完全に不可視。壊れた計画を静かに防ぐだけで、ユーザーに
  契約の存在を意識させない。

NIL / UNKNOWN / error の区別（§5）も、L0 ではメッセージの**トーンの違い**として現れ、
専門語としては現れない。

## 3. 案1：実行結果と翻訳を自然言語へ射影する

### 原則：説明を「生成」せず、構造化診断を「射影」する

`rust/src/interpreter/debug_diagnosis.rs` の `DebugDiagnosis` と
`AiDiagnosticPayload` は、説明に必要な材料をすべて構造化済みで保持する
（`when` `where_` `why` `summary` `evidence` `next_checks` `agreed_prefix` /
`recoverability` `semantic_area` `semantic_role` `nil_reason` `truth_value` `effect`）。

したがって自然言語の説明文は、**これらの enum・フィールドを文へ展開する決定的な
射影**で作る。LLM に「何が起きたか考えさせる」必要がないため、幻覚が構造的に排除される。

### `recoverability` を「次の一手」の文へ射影する

`recoverability_for()` が返す値はそのまま行動指示の語彙である。L0 向けの平文射影例：

| recoverability | L0 説明（射影） |
|---|---|
| `fixInput` | 「入力値が範囲外のようです。渡した値を見直してください。」 |
| `fixProgram` | 「手順が意図と噛み合っていません。」 |
| `addBudgetOrFixRecursion` | 「処理の上限に達しました。範囲を絞るか上限を増やしてください。」 |
| `handleUnknownOrNil` | 「答えが得られませんでした。既定値を決めますか？」 |
| `fixCapabilityOrForce` | 「この操作には許可が必要です。」 |
| `fixHost` | 「実行環境（入出力など）側の問題です。」 |

`next_checks`（`{label, detail}` 群）はすでに実行可能なステップの並びであり、L2 の
「詳しく」表示でそのまま箇条書きに射影できる。

### exactness が説明を強くする

数値が連分数の exact real なので、説明は「およそ1」ではなく「ちょうど 1/1」と言い切れる。
浮動小数の丸めが介在していないことを**真として**説明できる。

### i18n がほぼタダになる

射影元フィールドは言語中立（enum + 数値）なので、同じ診断を日本語・英語などへ射影できる。
NL 説明レイヤの多言語化は射影テーブルの差し替えで済む。

### 実装の形

新しい言語機能ではなく、既存 `--json` 出力（`agent-cli-output-contract.md`）の上に
乗る別レンダラとする。二段構え：(1) 決定的テンプレート射影（必ず正しい・安い）、
(2) 任意の LLM 整形（「フィールドを言い換えるだけ。新事実を足さない」と制約）。
(1) が常に正しい説明を保証し、(2) は読みやすさのみを担う。

## 4. 案2（軽量版）：契約で「壊れた計画」を実行前に弾く

### 軽量版の定義

案2には重い版（契約レジストリを**探索空間**として語を選ぶ）と軽い版（LLM が語を生成し、
契約は**事後検査**に使う）がある。**本層では軽量版を採用する。**

| | 重い版（不採用） | 軽量版（採用） |
|---|---|---|
| 語の選択 | 契約レジストリから探索 | LLM が生成 |
| 契約の役割 | 探索空間そのもの | 生成後の検査のみ |
| 実装コスト | 高い | 低い（既存の検査機構の再利用） |

### 検査に使う既存構造

- **フロー質量保存（§13）**：組んだ語の並びの consumes/produces 整合を実行前に検査。
  `mass: Fixed { consumes, produces }` を持つ語については静的に流量が合うかを判定でき、
  `Dynamic` 語については検査を健全に abstain する。
- **§7.14 契約**：`partiality`（Total/Partial/Projecting）と `nil_policy`
  （Passthrough/CreatesNil/RejectsNil/ConsumesNil/PreservesReason）を照合し、
  「NIL を生みうる語の後に fallback 口があるか」「NIL を拒否する語に NIL が流れていないか」
  を点検する。

### 学習曲線への影響：ゼロ

軽量版の検査は完全に不可視である。ユーザーは契約の存在を知らない。検査が通れば何も
起きず、壊れた計画は静かに案4へ回る（複数候補が成立 → 逆質問、成立する計画が無い →
平易な言葉での差し戻し）。将来、逆質問の質が物足りなくなった時点で重い版へ寄せる
余地は残すが、本層の初期スコープには含めない。

## 5. 案3：操作意図を修飾子の有限格子へ分類する

### 強みは「ターゲットが閉じた小さな空間」であること

自然言語の操作意図は無限だが、Ajisai の操作系は閉じている。

```
Target:      TOP（先頭点）   |  STAK（スタック全体）
Consumption: EAT（消費・既定） |  KEEP（残す＝分岐 bifurcation §13.2）
Fallback:    ^ VENT（bubble を既定値へ置換）
```

`TOP/STAK × EAT/KEEP` の 4 状態 + VENT しかない。NL→任意コード生成は検証不能だが、
**NL→修飾子は「有限格子への分類問題」**になり、生成よりはるかに信頼でき、かつ検査できる。

### 意図 → 修飾子のマッピング（内部表。ユーザーには見せない）

| 自然言語の言い回し | 落ちる先 |
|---|---|
| 「直前の値に」「先頭だけ」 | `TOP` |
| 「これまで全体に」「スタック全体を」 | `STAK` |
| 「使い切って」「消費して」 | `EAT`（既定） |
| 「元は残して」「コピーして別途」 | `KEEP` |
| 「失敗したら〜にする」「ダメなら既定値で」 | `^`(VENT) |

### 学習曲線への影響：既定で隠す

L0 では `EAT` / `TOP` を既定とし、操作様態を一切問わない。明示的な手掛かり
（「残して」等）があるときだけ `KEEP` 等へ振る。`KEEP` を推定したときは、フロー質量
保存で分岐が成立するか実行前に裏を取る（案2軽量版と共有）。様態が一意に決まらない
ときは案4へ委譲する。

## 6. 案4：UNKNOWN を「逆質問」のトリガーにする

### 2 つの発生源

- **実行時 UNKNOWN（実装済み）**：CF 比較が予算切れになると
  `comparison_unknown(word, agreed_prefix)` が `truthValue=unknown` + `agreedPrefix=N`
  を返す。L1 では「まだ決めきれていません。もっと詳しく調べますか？」と平易に提示する。
  `agreedPrefix` は内部的な**進捗指標**として、調べ直す価値があるかの判断に使う
  （文面には出さない）。深掘りの実体は `COMPARE-WITHIN` による予算拡大。
- **計画時 UNKNOWN（本層で新設）**：NL→計画の翻訳器が候補を一つに絞れないとき。
  **これを生成するのが案2軽量版の検査結果**である。契約照合の結果が
  「複数候補が成立」なら UNKNOWN として逆質問し、「成立する計画が無い」なら error 相当
  として平易に差し戻す。推測で一つの解釈に倒す（黙って FALSE 相当の決定をする）ことを
  禁ずる。

### NIL / UNKNOWN / error で UX のトーンを変える（用語は出さない）

| 内部状態 | 意味 | L0 で出すトーン |
|---|---|---|
| NIL（Bubble） | 答えが存在しない | 「値がありませんでした。既定値を決めますか？」 |
| UNKNOWN（Stagnation） | 答えはあるが未決定 | 「まだ決めきれていません。もっと調べますか？」 |
| error（Channel error） | 問いの立て方が壊れている | 「このままでは進めません。〜を直してください。」 |

「分からない」を全部 error にすると使いにくく、全部 null にすると危険、という問題への
Ajisai の構造的な答えがこの 3 分岐である。`nil_policy` が「どの語が NIL を生みうるか」を
契約として持つため、サーフェス側は fallback 口を**事前に**用意できる。

### Strong Kleene で逆質問を最小化する

三値論理（§7.5）の `TRUE OR UNKNOWN = TRUE` / `FALSE AND UNKNOWN = FALSE` により、
計画の一部が UNKNOWN でも結論が確定する場合は質問しない。**結論を左右する曖昧さだけを
聞く**。UNKNOWN の伝播がそのまま「どの曖昧さが本当に重要か」のフィルタになる。これは
学習曲線の観点でも重要で、ユーザーへの逆質問回数を最小化する。

## 7. アーキテクチャ上の規律

1. **本層は別アーティファクトとする。** `SPECIFICATION.html` には入れない（肥大化禁止）。
   サーフェス層の実装は CLI（`docs/dev/ai-first-competitive-upgrade-instructions.md`
   Phase 1）と word-manifest / 契約レジストリの上に乗る。
2. **意味論を変えない。** すべて observational / additive。既存プログラムの出力・
   NIL/UNKNOWN 挙動・Fraction の exactness を変えない。
3. **正典の重複を作らない。** 修飾子の表面構文・alias・契約は、tokenizer /
   `core_word_aliases.rs` / `word-manifest.json` / 契約レジストリを正典とし、本層が
   再パース・手書き複製しない（`human-surface-blackbox-instruction-review.md` 問題点7
   の轍を踏まない）。
4. **短さより意味保存を優先する。** 価値は記述量削減ではなく exactness と
   diagnosability にある。

## 8. 段階的な進め方（提案）

依存順。各段は独立した PR にできる。

1. **案1（決定的射影レイヤ）** — 最初の一歩。CLI `--json` の `DebugDiagnosis` /
   `AiDiagnosticPayload` → 自然言語のマッピング表を実装。L0 平文 + L2 詳細の二段。
   前向き翻訳がまだ無くても、実行結果の説明として単独で価値がある。
   **実装済み（第一段）**: `rust/src/cli/explain.rs`。`ajisai run|check --explain
   [--lang ja|en]` で、診断を平文へ決定的に射影する（Stagnation/Bubble/Channel
   error をトーンで区別、`recoverability` を次の一手の文へ、`nextChecks` を L2 詳細へ）。
   `--json` には追加フィールド `explanation` として出力（契約は
   `docs/dev/agent-cli-output-contract.md` §10）。成功実行中に生じた NIL バブルも
   射影する。LLM を介さない純射影で、幻覚は構造的に排除される。
2. **案3（修飾子推定）+ 案2軽量版（契約・質量検査）** — NL 操作意図 → 修飾子格子の
   分類器と、フロー質量保存 / §7.14 照合による事前検査。両者は計画生成で一体。
3. **案4（逆質問）** — 案2軽量版の検査結果（複数候補成立 / 成立なし）と実行時 UNKNOWN を
   L1 の平易な逆質問へ射影。Strong Kleene による質問最小化を含む。
4. **案1 を前向き翻訳へ拡張** — 「なぜこの語・この様態を選んだか」を契約に基づいて
   L2 で説明（ループ両端の説明可能性の完成）。

## 9. Non-goals / やってはいけないこと

- 自然言語を**直接実行**する処理系にすること（本層は翻訳・正規化層であって実行系ではない）。
- 表面構文を自然文へ寄せて言語本体を曖昧化すること（二層性を保つ）。
- `SPECIFICATION.html` を肥大化させること。
- 既存プログラムの出力・exactness・NIL/UNKNOWN 挙動を変えること。
- ユーザーに修飾子・契約・UNKNOWN・連分数の知識を要求すること（L0 で機構を露出しない）。
- 重い契約駆動探索を初期スコープで実装すること（案2は軽量版に限定）。

## 10. 期待する最終状態

```
- ユーザーは仕組みを知らずに L0 で要求を書き、平易な言葉で結果と次の一手を受け取る。
- 意図が一意に決まらないときだけ、平易な逆質問（L1）が出る。用語は出ない。
- 「詳しく」を選んだユーザーにだけ、生成された Ajisai・構造化診断・修飾子（L2）が開く。
- 壊れた計画は契約とフロー質量保存で実行前に静かに弾かれる。
- 説明はすべて実行時診断・契約に基づき、幻覚を含まない。exact な値は exact に説明される。
- 本層は SPECIFICATION.html の意味論を一切変えていない。
```
