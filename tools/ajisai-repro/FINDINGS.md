# Ajisai 仕様書からの独立再現と原典実装の比較

**日付**: 2026-06-30
**対象正典**: `SPECIFICATION.html`（Status: Canonical, Version: 2026-06-11）
**原典実装**: Rust コア（`rust/`）のヘッドレス CLI `ajisai run <file> --json`
**再現版**: `tools/ajisai-repro/ajisai.py`（Python。`SPECIFICATION.html` の散文のみから実装。Rust ソースは参照していない）

## 1. 方法

1. 正典 `SPECIFICATION.html` を読み、ホスト非依存の Core Profile（字句解析・値モデル・スタック・修飾子・算術/比較/論理/ベクタ/高階語・NIL/Bubble・表示）を **散文の記述だけ** から Python で再実装した（`ajisai.py`）。実装の Rust を読んでバイアスがかからないよう、原典コードは比較時まで参照していない。
2. 原典実装は `cargo build --bin ajisai --release` でビルドした headless CLI を使い、`--json` で機械可読なスタック/出力を得た（`probe.py`）。
3. 同一プログラム 79 本を両者に通し、観測（最終スタックの表示文字列・出力・エラー種別）を突き合わせた（`compare.py`）。
4. 差異が出たものは、正典の **どの条項が決め手か** を確認して分類した。

最終結果: **79 本中 64 本一致・15 本相違**（`compare-output.txt`）。
相違は下記のとおり、単なる実装バグというより **正典そのものの作り込みの甘さ／正典内部の矛盾** に起因するものが中心だった。

## 2. もっとも重い発見：正典内部の矛盾（散文 vs 適合性スイート）

`SPECIFICATION.html` は権威順位（§2.5）で「散文 → 数学的形式化 → **適合性スイート `tests/conformance/`** → …」を canonical と定める。ところがこの正典内の二層が**互いに矛盾**している。原典実装は散文ではなく **適合性スイート側** に従っており、散文に忠実な再現版だけが両者から外れる。

### 2.1 `STAK`（`..`）修飾子の意味が二重定義

| 出典 | 規定 |
|---|---|
| 散文 §6.1 | 「`STAK` … スタック全体が被演算子として扱われる」（カウント引数なし） |
| 適合性スイート case 42 | `1 2 3 3 STAK ADD => 6/1`（**先頭にカウント `3` を置く**） |

原典実装の挙動（確認済み）:

```
1 2 3 STAK ADD        -> stackUnderflow      # 散文どおり「スタック全体」では動かない
1 2 3 3 STAK ADD      -> 6/1                 # スイートどおり「カウント N を取り N 個畳み込む」
5 4 3 2 1 5 STAK LT   -> FALSE               # 先頭の 5 がカウント、残り5個が狭義単調増加か判定
```

- 散文 §6.1／§7.4（`STAK` モードの並び性質）は **カウント引数の存在を一言も書いていない**。さらに二項語 `ADD` が「スタック全体」をどう畳み込むのか（fold なのか）すら散文には定義がない。
- スイートと実装は「カウント前置の畳み込み」という**別機構**でこれを解決している。
- 結果として、散文だけ読んで作った再現版は `1 2 3 STAK ADD` を「全体」と解釈して実装と食い違う。**これは実装バグではなく、正典の二層が食い違っているために生じる不可避の乖離。**

### 2.2 `GET` / `LENGTH` は元ベクタを消費しない（§6.2 と矛盾）

| 出典 | 規定 |
|---|---|
| 散文 §6.2 | 「`EAT`（既定）… 演算後に被演算子をスタックから取り除く」 |
| 適合性スイート case 16 | `[ 10 20 30 ] [ 0 ] GET => [ 10/1 20/1 30/1 ] 10/1`（**元ベクタが残る**） |
| 適合性スイート case 20 | `[ 1 2 3 ] LENGTH => [ 1/1 2/1 3/1 ] 3/1`（**元ベクタが残る**） |

原典実装は適合性スイートどおり元ベクタを残す。一方で `REVERSE` `REMOVE` `REPLACE` `CONCAT` は元ベクタを **消費** する。つまり:

- 既定 `EAT` の散文規定（§6.2）と、`GET`/`LENGTH` が被演算子を残す挙動が矛盾。
- しかも実装内部でも非一貫（`GET`/`LENGTH` は残す、`REVERSE`/`REMOVE` は消費）。どの語が「ソース文脈を残す」のかを正典散文は定義していない。

### 2.3 `GET` のインデックス引数はベクタ包み（§7.1 と不一致）

スイート case 15/16 は `[ 10 20 30 ] [ 0 ] GET`（インデックスを `[ 0 ]` と**ベクタで包む**）。散文 §7.1 は「`GET` — 与えられたインデックスの要素を取得」とだけ書き、`[index]` 包みを規定していない。原典実装はベクタ包みを要求し、素の `[ 1 2 3 ] 0 GET` でも（インデックスを単スカラとして）通るが、`REPLACE`/`INSERT` は `[index element]` 包みを **必須** とする（後述 §4）。

## 3. 実装が散文から逸脱している箇所（スイートが沈黙）

### 3.1 正典 §2.3.1.1 の「ユーザが頼ってよい」例が動かない

§2.3.1.1 は次を「表現や比較予算を知らなくてもユーザが頼れる」例として明示する:

```
2 MATH@SQRT 2 MATH@SQRT SUB 0 EQ   （期待: TRUE）
```

原典実装での結果: **`unknownWord` エラー**。修飾名 `MATH@WORD` が、事前に `'MATH' IMPORT` していないと解決されない。

- §7 は「`MODULE@WORD` はそのモジュールの canonical エントリのみに解決される」、§7.14 は「canonical_home … は `MODULE@WORD` として解決できる唯一のモジュール」と規定。修飾名は **インポート状態に依らず常に解決される** のが修飾名の存在意義であり、§2.3.1.1 の例も IMPORT を伴わずに提示されている。
- 実装は `'MATH' IMPORT` 後でなければ `MATH@SQRT` を解決しない。**正典が看板に掲げる例が、原典実装で再現できない。** 最も象徴的な乖離。

### 3.2 `CHR` の範囲外コードポイントが NIL ではなく空文字列

| 入力 | 散文 §11.2 / §7.6 の規定 | 原典実装 | 再現版 |
|---|---|---|---|
| `-1 CHR` | （整数だが範囲外）→ NIL `invalidEncoding` | `''`（空文字列） | `NIL` |
| `55296 CHR`（サロゲート D800） | 範囲外 → NIL `invalidEncoding` | `''` | `NIL` |
| `1114112 CHR`（0x110000, 上限超過） | 範囲外 → NIL `invalidEncoding` | `''` | `NIL` |

§11.2 は「数値コードポイントが有効な Unicode スカラ範囲外 → `NilReason::InvalidEncoding`（= NIL）」と明記。原典実装は **NIL を作らず空文字列を返す**。Bubble ルール（§11.2）・NIL reason 被覆テスト（§15.2）の要求に反する明確な逸脱。

### 3.3 `COLLECT` がカウント引数を要求（§7.1 と矛盾）

散文 §7.1: 「`COLLECT` — 現在の **全** スタック値を 1 本のベクタに集める」。
原典実装: `1 2 3 COLLECT -> stackUnderflow`、`1 2 3 3 COLLECT -> [1 2 3]`。つまり **カウント `n` を要求** する。「全スタック値」という散文規定と矛盾。

### 3.4 文字列 `CONCAT` が Text ロールを失いコードポイント数値ベクタへ崩落

`'ab' 'cd' CONCAT -> [ 97/1 98/1 99/1 100/1 ]`。§12.2 は「コレクション内の各要素は各々のロールでレンダされ、ネストした Text 要素は引用符付き文字列のまま」「codepoint fraction に崩落しない」と規定するが、文字列連結結果が Text ロールを失い裸の数値ベクタになっている。散文は CONCAT が文字列を受けるか自体を明記しておらず、結果のロール保持規定（§12.2）と衝突。

## 4. 正典の作り込みの甘さ（仕様の穴：署名・端数規則の未定義）

これらは「実装が間違い」というより **正典が決めていない** ため、散文に忠実な再現版と実装が割れた箇所。

| 項目 | 散文の記述 | 欠けている規定 | 実装の選択 |
|---|---|---|---|
| `RANGE` | 「start から end まで（任意で step）の整数列」 | スタック署名（`[start end]` ベクタ包み）と端点の包含/排他 | `[ 1 5 ] RANGE => [1 2 3 4 5]`（**end 包含**、ベクタ引数） |
| `REPLACE` / `INSERT` | 「インデックスの要素を置換/挿入」 | スタック署名 | `vector [index element]`（2要素ベクタ必須）。`vector index element` は `custom` エラー |
| `MOD` by 0 | §7.3 は FLOOR ベース剰余、§11.2 は **DIV** のゼロ除算のみ NIL 化 | `MOD` のゼロ除算の扱い | `5 0 MOD -> custom` エラー（DIV は NIL なのに非対称） |
| `ROUND` 端数 | 「最も近い整数に丸める」 | タイ（0.5）の丸め方向 | `-2.5 -> -3`, `0.5 -> 1`（**ゼロから遠い方**）。なお §4.2.5 の NICF は「half-down」と規定しており、丸め規約が語間で非一貫 |

特に `MOD`-by-zero は、§11.2 が「DIV のゼロ除算 → NIL」を Bubble ルールの代表例として挙げながら、同じくゼロ除算を内包する `MOD` を列挙していない典型的な穴。NIL 透過の対称性（§7.12 で `MOD` も NIL-passthrough に列挙）からは NIL 化が自然だが、実装は error を返す。

## 5. 一致した領域（正典が機能している部分）

64/79 本は完全一致。とくに以下は散文どおりに両者が一致し、正典がよく作り込まれていることを示す:

- 厳密有理数の算術と表示（`1 2 ADD => 3/1`, `10 3 DIV => 10/3`、浮動小数を一切経由しない `num/den` 表示）。
- `SQRT` の連分数表示と **厳密代数比較**（`2 SQRT 2 SQRT SUB => 0/1`、`2 SQRT 2 SQRT EQ => TRUE`）。
- 三値論理 K3（`TRUE FALSE AND => FALSE` 等）と Boolean とスカラの区別（`TRUE 1 EQ => FALSE`、§4.1 どおり）。
- Bubble/NIL と `^`(VENT) コアレッシング（`5 0 DIV => NIL`、`5 0 DIV 99 ^ => 99`、`5 0 DIV 1 ADD => NIL` の NIL 透過）。
- 出力境界の引用符除去（`'TEST' PRINT => TEST`、`[ 'AB' 'CD' ] PRINT => [ 'AB' 'CD' ]`、§7.9）。

## 6. 結論

ユーザの仮説――「差異が出るなら、原典 Ajisai の仕様の作り込みが甘いか、仕様と実装に乖離がある」――は、両方とも実証された。

1. **正典内部の矛盾（最重）**: `SPECIFICATION.html` の散文（§6.1 STAK、§6.2 既定 EAT、§7.1 GET 署名）が、同じく canonical な `tests/conformance/` の期待値と真っ向から食い違う。原典実装は散文ではなく**スイート側に従う**。すなわち実装は「実装に最も近い正典層（スイート）」には適合するが、最上位権威の散文には適合しない。
2. **実装の散文逸脱（スイート未被覆）**: `MATH@SQRT` の修飾名解決（§2.3.1.1 の看板例が動かない）、`CHR` 範囲外の空文字列化（§11.2 違反）、`COLLECT` のカウント要求（§7.1 違反）など、散文が明確に決めているのに実装が外れている箇所がある。これらはスイートにテストが無く、ガードされていない。
3. **正典の穴（作り込みの甘さ）**: `RANGE`/`REPLACE`/`INSERT` のスタック署名、`RANGE` の端点包含、`MOD`-by-zero、`ROUND` の端数規則が散文に欠落。散文だけからの忠実な移植は、実装と必然的に割れる。

総じて、**`SPECIFICATION.html` 単体では実装非依存の同一言語を再現できない**（§2.4 が掲げる目標「Rust/WASM/TS を参照せずに同じ言語を再現できる」を満たさない）。同一性は実質的に適合性スイート＋実装が補完しており、散文はそれらと矛盾・不足を抱えている。

## 7. 再現物

- `ajisai.py` — 散文のみから実装した Python 再現版（Core Profile 中心）。
- `probe.py` — 原典 CLI 実行ヘルパ。
- `compare.py` — 79 本のバッテリで両者を突き合わせる比較ハーネス。
- `compare-output.txt` — 直近の比較結果。

再実行:
```
cargo build --bin ajisai --release      # rust/ 配下
python3 tools/ajisai-repro/compare.py
```
