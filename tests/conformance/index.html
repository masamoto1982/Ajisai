Ajisai Conformance Suite

This file defines the phenomenon of Ajisai language-independently. An implementation is an Ajisai implementation if and only if it reproduces every case below (see PORTABILITY.md and SPECIFICATION.html § "Conformance and Identity").

Contract

  1. Each case is a <section class="ajisai-case"> with an id and a data-category.
  2. ajisai-source is run as an Ajisai program.
  3. The observed final result — the whole stack, each value rendered value-by-value — must equal ajisai-expect-result after normalization. A case may instead include ajisai-expect-error; then execution must fail with an error containing that text, and the stack result is not compared.
  4. The fired host effects must equal the ordered list of ajisai-effect elements inside ajisai-expect-effects: both data-kind and data-payload, in order. The observation target is the structured host-effect channel, never the human-readable output buffer. An empty ajisai-expect-effects means no effects.
  5. Hosted cases may request a deterministic host using data-host-now-millis, data-host-random-hex, and data-host-capabilities. Capability names use protocol spelling (clock, secureRandom, serial, audio, jsonExport, config, effect).

Normalization (minimal, strict-leaning)

Only whitespace between tokens (spaces, tabs, newlines) is normalized, because whitespace is a separator in Ajisai's value model, not a value. Everything else is matched exactly: numeric notation (every number is a reduced numerator/denominator, integers included — 3 renders as 3/1), element order, nesting, and the spellings NIL and UNKNOWN. This line may only be loosened in future, never tightened, and any change must record why.

Note: Ajisai has no empty-vector literal — [ ] is rejected by the value model ("Use NIL for empty values"). NIL is the canonical absence, so there is no "empty vector vs NIL" case to encode here; the distinction does not arise as an observable literal.
